Architecture — Ladder Room Online

Version v1.1 — 2026-04-19 — Based on EDD v1.1, PRD v1.1, PDD v2.1

1. 架構概覽

Ladder Room Online 採用 Clean Architecture 分層設計，核心原則為依賴方向永遠指向內層 (Domain)，外層 (Infrastructure/Presentation) 依賴抽象介面而非具體實作。

graph TD
    subgraph shared["packages/shared (前後端共用)"]
        D["Domain Layer\nRoom · Player · Ladder · DomainError\n純 Entity / Value Object"]
        UC["Use Cases Layer\nGenerateLadder · ValidateGameStart · ComputeResults\n純函式，零 I/O"]
        D --> UC
    end

    subgraph server["packages/server"]
        APP["Application Layer\nRoomService · GameService\nWsMessageHandler · PubSubHandler"]
        INF["Infrastructure Layer\nRedisClient · RoomRepository · PubSubBroker\nWsServer · WsSession"]
        PRES["Presentation Layer\nFastify routes · AJV schemas\nHTTP 分派 / WS 升級驗證"]
    end

    UC --> APP
    APP --> INF
    APP --> PRES
層級所在套件職責外部依賴
Domainpackages/sharedEntity、Value Object、純業務規則、DomainError無
Use Casespackages/shared協調 Domain 物件完成業務流程，回傳純資料結構僅依賴 Domain
Applicationpackages/server/src/application呼叫 Use Cases、協調 Repository 介面、發布 WS 事件Use Cases + 抽象 Interface
Infrastructurepackages/server/src/infrastructureRedis 實作、WebSocket 封裝、ioredis / jose SDK實作 Application Interface
Presentationpackages/server/src/presentationHTTP Route、AJV Schema 驗證、WS 訊息分派Application Service
黃金法則： 任何模組只能 import 同層或內層的模組；packages/shared 不得 import 任何 Node.js 內建 I/O 模組（fs、net、http 等）。

2.1 packages/shared — 輸出清單

路徑輸出內容限制
src/domain/entities/Room, Player, Ladder, LadderSegment純 TypeScript class/interface
src/domain/value-objects/RoomCode, RoomStatus純函式驗證
src/domain/errors/DomainError 基類及子類無外部依賴
src/use-cases/GenerateLadder, ValidateGameStart, ComputeResults純函式，可在瀏覽器執行
src/prng/djb2, createMulberry32, fisherYatesShuffle無 Math.random()
src/types/index.ts所有共用 TypeScript interface/type不含 class 實作
  • packages/client 可 import shared 的全部匯出
  • packages/server 可 import shared 的全部匯出
  • packages/shared 不得 import client 或 server 的任何模組
  • client 與 server 之間無直接 import（透過 WS/HTTP 通訊）

4. 網路設計

端口與協議

3000HTTP + WSFastify 服務（容器內）
6379Redis protocolRedis 服務（集群內）

Sticky Session 策略

Nginx Ingress 使用 nginx.ingress.kubernetes.io/affinity: cookie 或 ip_hash，確保同一房間的 WebSocket 連線路由至同一 Pod，避免跨 Pod 廣播的複雜性。跨 Pod 廣播透過 Redis Pub/Sub 處理。

5. 韌性設計

場景對策恢復機制
Pod 意外重啟房間狀態存於 Redis，不依賴 Pod 記憶體客戶端自動重連，從 Redis 恢復 ROOM_STATE
客戶端斷線isOnline=false 標記，60s 內重連可恢復JWT token 重連，伺服器發送 ROOM_STATE_FULL
Redis 主節點故障Redis Sentinel / Replica 自動 failoverPod 重連至新主節點，最多 30s 中斷
所有玩家離線最後 WS 關閉時 EXPIRE 5min玩家 5 分鐘內重連可恢復房間
競態條件（狀態轉換）Redis WATCH + MULTI/EXEC 樂觀鎖失敗時返回錯誤，客戶端重試
