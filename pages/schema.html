SCHEMA — Redis Schema Design Document

Redis Schema — Ladder Room Online

Redis is the sole persistence layer. There is no SQL database. All state lives in Redis key-value structures with explicit TTLs.

1. 資料儲存概覽

Ladder Room Online 使用 Redis 作為唯一的持久層。選擇原因：

  • 房間生命週期短（數分鐘到數小時），天然適合 TTL 驅動的資料管理
  • WebSocket 廣播需要低延遲讀寫，Redis 的亞毫秒響應完全符合
  • 原子操作（INCR、SETNX、MULTI/EXEC、WATCH）消除競態條件
  • 記憶體用量可預測（每房間 ~88 KB，100 房間僅 ~8.8 MB）
類別儲存內容Redis 型別
房間狀態status、hostId、winnerCount、revealMode、players[]String (JSON)
梯子資料seed、seedSource、rowCount、segments[]、results[]String (JSON)
揭示計數revealedCount 原子計數器String (integer)
踢除名單kickedPlayerIds SetSet
Session 對應playerId → sessionId HashHash

2. Redis Key Schema

Key 模式TypeValueTTL用途
room:{code}String (JSON)序列化的 Room 物件（含 players 陣列）24h房間主狀態
room:{code}:ladderString (JSON)LadderData JSON（seed、segments、results）與主鍵同步梯子結構與結果
room:{code}:revealedCountString (counter)整數字串，如 "0" 到 "N"與主鍵同步揭示進度原子計數器
room:{code}:kickedSet{ playerId1, playerId2, ... }與主鍵同步踢除玩家 ID 集合
room:{code}:sessionsHash{ playerId: sessionId }與主鍵同步追蹤活躍 WebSocket Session

範例鍵（房間碼 ALPHA1）：

room:ALPHA1
room:ALPHA1:ladder
room:ALPHA1:revealedCount
room:ALPHA1:kicked
room:ALPHA1:sessions

5. TTL 策略

房間狀態TTL觸發條件
waiting / running / revealing24h (86400s)建立或任何狀態變更時重設
finished1h (3600s)轉換至 finished 狀態時設定
所有玩家離線5min (300s)最後一個 WS 連線關閉時 EXPIRE
Pod 重啟維持現有 TTL無連線的房間不重設，自然過期

所有 room:{code}:* 子鍵與主鍵使用相同的 TTL，並在主鍵更新時使用 MULTI/EXEC 批次重設所有子鍵的 EXPIRE。
