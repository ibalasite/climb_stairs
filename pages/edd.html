EDD — Ladder Room Online

Version v1.1 — 2026-04-19 — Clean Architecture Monorepo

1. 系統概覽

Ladder Room Online 是一款基於 HTML5 Canvas 的多人線上爬樓梯抽獎系統，採用 WebSocket 長連接驅動即時遊戲狀態同步，支援最多 50 名玩家共享同一房間。後端以 Fastify 處理 HTTP REST 操作，ws 原生 WebSocket 處理即時通訊，Redis 同時承擔分散式狀態鎖（原子操作）、房間資料持久化與跨 Pod Pub/Sub 廣播的角色；前端以 Vanilla TypeScript + Vite 建構，透過 HTML5 Canvas 逐段繪製梯子揭示動畫，全程無任何 UI 框架依賴。

整體系統遵循 Clean Architecture 分層原則，核心遊戲邏輯（PRNG、狀態機、梯子生成）封裝於 packages/shared 並在前後端共用，保證演算法一致性可驗證。部署層透過 Kubernetes HPA 依 WebSocket 連線數自動水平擴展後端 Pod，Nginx Ingress 做 sticky session 確保同一房間玩家路由至同一 Pod，Redis 作為唯一共享狀態層解耦 Pod 間狀態依賴。

2. 架構設計

2.1 Clean Architecture (SOLID + 分層 + DI)

層級職責依賴方向
Domain (shared)Entity、Value Object、純業務規則、DomainError無外部依賴
Use Cases (shared)協調 Domain 物件完成業務流程，返回純資料結構僅依賴 Domain
Application (server)呼叫 Use Cases、協調 Repository、發布 WS 事件依賴 Use Cases + Interfaces
Infrastructure (server)Redis 實作、WebSocket 封裝、外部 SDK實作 Application 定義的 Interface
Presentation (server)HTTP Route、Schema 驗證、WS 訊息分派依賴 Application Service

DI 策略：constructor injection，container.ts 以工廠函式組裝所有依賴。測試時直接傳入 mock 實作，無需 DI 框架。

3. Room 狀態機

房間有 4 個核心狀態，轉換由主持人操作觸發，系統不允許跳過或回滾（除 reset 路徑外）。

stateDiagram-v2
    [*] --> waiting: Room Created
    waiting --> running: START_GAME\n(N≥2, W set)
    running --> revealing: BEGIN_REVEAL
    revealing --> finished: All Revealed
    finished --> waiting: RESET_ROOM\n(play again)
    waiting --> [*]: Room Expired (TTL)
State描述允許操作
waiting等待玩家加入，可設定中獎名額加入、踢除、設定 W、開始遊戲
running樓梯已生成，可進入揭曉開始揭曉
revealing逐步或自動揭曉路徑下一位、全部揭曉、切換模式
finished所有路徑揭曉完畢再玩一局（reset）

4. 核心資料模型

Room

interface Room {
  code: string;           // 6-char Room Code
  status: RoomStatus;     // 'waiting' | 'running' | 'revealing' | 'finished'
  hostId: string;         // playerId of the host
  winnerCount: number | null; // W (null until set)
  revealMode: 'manual' | 'auto';
  autoRevealIntervalSec: number | null; // 1-30
  kickedPlayerIds: string[];
  createdAt: number;      // Unix ms
  updatedAt: number;      // Unix ms
  players: Player[];
}

Player

interface Player {
  id: string;         // UUID v4
  nickname: string;   // 1-20 Unicode chars
  colorIndex: number; // 0-based palette index
  isHost: boolean;
  isOnline: boolean;
}

ResultSlot

interface ResultSlot {
  playerId: string;
  startCol: number;  // top column index (0-based)
  endCol: number;    // bottom column index
  isWinner: boolean;
}

LadderData

interface LadderData {
  seedSource: string;    // UUID hex used as djb2 input
  seed: number;          // djb2(seedSource) result
  rowCount: number;      // clamp(N*3, 20, 60)
  colCount: number;      // = N (one column per player)
  segments: LadderSegment[]; // horizontal rungs
  results: ResultSlot[]; // computed path outcomes
}

5. PRNG 算法

The ladder generation pipeline is deterministic, reproducible, and verifiable by any party with the seed.

// Step 1: djb2 hash — string → uint32
function djb2(str: string): number {
  let hash = 5381;
  for (let i = 0; i < str.length; i++) {
    hash = ((hash << 5) + hash) ^ str.charCodeAt(i);
    hash = hash >>> 0; // force uint32
  }
  return hash;
}

// Step 2: Mulberry32 — seeded PRNG → [0, 1) floats
function createMulberry32(seed: number): () => number {
  return function() {
    seed += 0x6D2B79F5;
    let t = seed;
    t = Math.imul(t ^ (t >>> 15), t | 1);
    t ^= t + Math.imul(t ^ (t >>> 7), t | 61);
    return ((t ^ (t >>> 14)) >>> 0) / 4294967296;
  };
}

// Step 3: Fisher-Yates shuffle — in-place, uses PRNG
function fisherYatesShuffle(arr: T[], rand: () => number): T[] {
  const a = [...arr]; // immutable — new copy
  for (let i = a.length - 1; i > 0; i--) {
    const j = Math.floor(rand() * (i + 1));
    [a[i], a[j]] = [a[j], a[i]];
  }
  return a;
}

The seedSource is a server-generated UUID hex string. djb2(seedSource) produces the numeric seed passed to createMulberry32. The same seed always produces the same ladder — making results tamper-proof and independently verifiable by any client.
