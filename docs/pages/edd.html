1. 系統概覽

Ladder Room Online 是一款基於 HTML5 Canvas 的多人線上爬樓梯抽獎系統，採用 WebSocket 長連接驅動即時遊戲狀態同步，支援最多 50 名玩家共享同一房間。後端以 Fastify 處理 HTTP REST 操作，ws 原生 WebSocket 處理即時通訊，Redis 同時承擔分散式狀態鎖（原子操作）、房間資料持久化與跨 Pod Pub/Sub 廣播的角色；前端以 Vanilla TypeScript + Vite 建構，透過 HTML5 Canvas 逐段繪製梯子揭示動畫，全程無任何 UI 框架依賴。

4. 主要設計決策

4.1 選用原生 ws 而非 Socket.IO

Socket.IO 的 polling fallback 在本專案中無需求（所有目標瀏覽器均支援原生 WebSocket）。ws 的 bundle size 遠小於 Socket.IO，降低 Node.js 冷啟動時間。跨語言客戶端直接使用標準 WS 協定無需額外 adapter。

4.2 Monorepo 結構

packages/shared 在前後端共用梯子生成算法（PRNG + GenerateLadder + ComputeResults），確保兩端結果可獨立驗證。單一 CI pipeline 同時執行跨套件類型檢查，及早捕捉介面不相容問題。

4.3 Redis Pub/Sub 用於跨 Pod 廣播

每個 Pod SUBSCRIBE room:*:events，任一 Pod 觸發事件時 PUBLISH 至對應頻道，所有 Pod 的 PubSubHandler 負責轉發至本地 WsSession。Redis 是唯一共享狀態層，不需要引入第三方訊息佇列。

4.4 不使用 ORM

直接使用 ioredis 命令，Room 以 JSON 字串儲存於單一 Redis key。Room 是聚合根，以 JSON 整包讀取/寫入更符合 Redis 用途。原子操作需直接控制 Redis 命令；ORM 抽象層反而降低可控性。

4.5 JWT HS256 作為 Session Token

Token payload：{ playerId, roomCode, role: "host" | "player", exp }。有效期：6 小時（21600 秒）。WS Upgrade 握手時驗證 JWT；一旦 WS 連線建立，後續訊息不重驗 exp（MVP 接受的安全取捨）。

6. 一致性保證

操作一致性機制
START_GAMEWATCH + MULTI/EXEC；EXEC null → 重試最多 3 次
BEGIN_REVEALLua Script 原子執行：GenerateLadder 計算 + ladder/results/status=revealing 寫入
END_GAMEWATCH + MULTI/EXEC：status → finished；廣播含 seed 的 ROOM_STATE
PLAY_AGAINWATCH + MULTI/EXEC 原子更新整包 Room JSON；清空 ladder/kickedPlayerIds
REVEAL_NEXT先 INCR 取得 newCount，再 WATCH + MULTI/EXEC 寫入 Room JSON 並 PUBLISH
玩家加入/離線Room JSON 整包 SET，配合 WATCH 防止 lost update
