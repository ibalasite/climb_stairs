Version: v1.2   Date: 2026-04-19   Based on: EDD v1.3, PRD v1.3, PDD v2.2

1. 架構概覽

Ladder Room Online 採用 Clean Architecture 分層設計，核心原則為依賴方向永遠指向內層 (Domain)，外層 (Infrastructure/Presentation) 依賴抽象介面而非具體實作。

        UC["Use Cases Layer\nGenerateLadder · ValidateGameStart · ComputeResults\n純函式，零 I/O"]
        D --> UC
    end

    subgraph server["packages/server"]
        APP["Application Layer\nRoomService · GameService\nWsMessageHandler · PubSubHandler\n協調 Use Cases + Repository"]
        INF["Infrastructure Layer\nRedisClient · RoomRepository · PubSubBroker\nWsServer · WsSession\n實作 Application 定義的 Interface"]
        PRES["Presentation Layer\nFastify routes · AJV schemas\nHTTP 分派 / WS 升級驗證"]
    end

    UC --> APP
    APP --> INF
    PRES --> APP
層級所在套件職責外部依賴
Domainpackages/sharedEntity、Value Object、純業務規則、DomainError無
Use Casespackages/shared協調 Domain 物件完成業務流程，回傳純資料結構僅依賴 Domain
Applicationpackages/server/src/application呼叫 Use Cases、協調 Repository 介面、發布 WS 事件Use Cases + 抽象 Interface
Infrastructurepackages/server/src/infrastructureRedis 實作、WebSocket 封裝、ioredis / jose SDK實作 Application Interface
Presentationpackages/server/src/presentationHTTP Route、AJV Schema 驗證、WS 訊息分派Application Service

黃金法則： 任何模組只能 import 同層或內層的模組；packages/shared 不得 import 任何 Node.js 內建 I/O 模組（fs、net、http 等）。

3. 模組職責

模組路徑職責
container.ts以工廠函式組裝所有依賴（DI 根），不含任何業務邏輯
main.ts啟動 Fastify、掛載 WsServer、初始化 PubSubBroker，設定 graceful shutdown
presentation/routes/rooms.ts處理房間 CRUD HTTP 路由
presentation/routes/players.ts處理玩家 HTTP 路由
application/services/RoomService.ts協調房間建立、加入、踢出等業務流程
application/services/GameService.ts協調遊戲開局、揭示、結束、再玩一局流程
application/handlers/WsMessageHandler.ts解析 ClientEnvelope，執行 role + room.hostId 驗證，分派至對應 Service
application/handlers/PubSubHandler.ts訂閱 Redis 頻道，轉發 PubSubMessage 至本地 WsSession
infrastructure/redis/RoomRepository.tsIRoomRepository 實作：Redis CRUD、TTL 管理、Lua Script
infrastructure/websocket/WsServer.ts封裝 ws.Server，處理 HTTP Upgrade、JWT 驗證、踢除攔截
infrastructure/websocket/WsSession.ts管理單一 WS 連線生命週期：心跳、序列化、速率限制（60 msg/min）

4. 主要設計決策

4.1 選用原生 ws 而非 Socket.IO

Socket.IO 的 polling fallback 在本專案中無需求。ws bundle size 遠小，降低冷啟動時間。跨語言客戶端直接使用標準 WS 協定。

4.2 Monorepo 結構

三套件 monorepo：packages/shared、packages/server、packages/client。shared 在前後端共用梯子生成算法，確保兩端結果可獨立驗證。

4.3 Redis Pub/Sub 用於跨 Pod 廣播

Redis 是唯一共享狀態層，Pub/Sub 不需要引入第三方訊息佇列即可完成 Pod 間廣播。room:*:events pattern subscription 確保新房間自動被訂閱。

4.4 不使用 ORM

直接使用 ioredis 命令，Room 以 JSON 字串儲存於單一 Redis key。Room 是聚合根，整包讀取/寫入符合 Redis 用途。原子操作需直接控制命令，ORM 抽象層反而降低可控性。

4.5 JWT HS256 (jose) 作為 Session Token

無狀態驗證：每個請求自帶身份資訊，不需額外 Redis 查詢。jose 支援 Web Crypto API。HS256 在 shared secret 模型下實作簡單。Token 有效期：6 小時（21600 秒）。

4.6 MVP 單實例，HPA 為 Post-MVP

MVP 部署單一 Node.js 實例。架構已預留多 Pod 設計（Pub/Sub、RoomRepository 無本地狀態）：Post-MVP 啟用 HPA 只需改 k8s manifest。

5. Import 規則

  • packages/client 可 import shared 的全部匯出
  • packages/server 可 import shared 的全部匯出
  • packages/shared 不得 import client 或 server 的任何模組
  • client 與 server 之間無直接 import（透過 WS/HTTP 通訊）
