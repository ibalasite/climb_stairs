1. 架構概覽

Ladder Room Online 採用 Clean Architecture 分層設計，核心原則為依賴方向永遠指向內層 (Domain)，外層 (Infrastructure/Presentation) 依賴抽象介面而非具體實作。

        UC["Use Cases Layer\nGenerateLadder · ValidateGameStart · ComputeResults\n純函式，零 I/O"]
        D --> UC
    end

    subgraph server["packages/server"]
        APP["Application Layer\nRoomService · GameService\nWsMessageHandler · PubSubHandler\n協調 Use Cases + Repository"]
        INF["Infrastructure Layer\nRedisClient · RoomRepository · PubSubBroker\nWsServer · WsSession\n實作 Application 定義的 Interface"]
        PRES["Presentation Layer\nFastify routes · AJV schemas\nHTTP 分派 / WS 升級驗證"]
    end

    UC --> APP
    APP --> INF
    PRES --> APP
層級所在套件職責外部依賴
Domainpackages/sharedEntity、Value Object、純業務規則、DomainError無
Use Casespackages/shared協調 Domain 物件完成業務流程，回傳純資料結構僅依賴 Domain
Applicationpackages/server/src/application呼叫 Use Cases、協調 Repository 介面、發布 WS 事件Use Cases + 抽象 Interface
Infrastructurepackages/server/src/infrastructureRedis 實作、WebSocket 封裝、ioredis / jose SDK實作 Application Interface
Presentationpackages/server/src/presentationHTTP Route、AJV Schema 驗證、WS 訊息分派Application Service
黃金法則： 任何模組只能 import 同層或內層的模組；packages/shared 不得 import 任何 Node.js 內建 I/O 模組。

3. 模組職責

模組路徑職責
container.ts以工廠函式組裝所有依賴（DI 根），回傳完整注入樹，不含業務邏輯
main.ts啟動 Fastify 伺服器、掛載 WsServer、初始化 PubSubBroker，設定 graceful shutdown
presentation/routes/rooms.ts處理房間 CRUD HTTP 路由（POST /rooms、GET /rooms/:code 等）
presentation/routes/players.ts處理玩家 HTTP 路由（POST /rooms/:code/players、DELETE /rooms/:code/players/:id）
application/services/RoomService.ts協調房間建立、加入、踢出等業務流程，呼叫 RoomRepository 並發布 ROOM_STATE 廣播
application/services/GameService.ts協調開局（START_GAME）、開始揭示（BEGIN_REVEAL，原子生成 LadderData）、REVEAL_NEXT、END_GAME、PLAY_AGAIN
application/handlers/WsMessageHandler.ts解析 ClientEnvelope，執行 role + room.hostId 驗證，分派至對應 Service 方法
application/handlers/PubSubHandler.ts訂閱 Redis room:*:events 頻道，轉發至房間內所有本地 WsSession
infrastructure/redis/RedisClient.ts建立並匯出 ioredis 單例（含連線重試設定）
infrastructure/redis/RoomRepository.ts實作 IRoomRepository：Room CRUD、TTL 管理；BEGIN_REVEAL 使用 Lua Script 原子寫入
infrastructure/redis/PubSubBroker.ts封裝 Redis PUBLISH 與 SUBSCRIBE，抽象跨 Pod 廣播細節
infrastructure/websocket/WsServer.ts封裝 ws.Server，處理 HTTP Upgrade、JWT 驗證、踢除攔截（close 4003），建立 WsSession
infrastructure/websocket/WsSession.ts管理單一 WS 連線生命週期：心跳、訊息序列化、速率限制（60 msg/min）、斷線計時

4. 主要設計決策

4.1 選用原生 ws 而非 Socket.IO

理由： Socket.IO 的 polling fallback 在本專案無需求；ws bundle size 遠小於 Socket.IO；跨語言客戶端可直接使用標準 WS 協定。

取捨： 需自行實作房間廣播、心跳、重連邏輯；透過 WsSession 封裝後複雜度可控。

4.2 Monorepo 結構

理由： packages/shared 在前後端共用梯子生成算法，確保兩端結果可獨立驗證；單一 CI pipeline 跨套件類型檢查；tsconfig 互相引用，型別變更即時傳播。

取捨： npm workspace 增加初始設定複雜度；Turborepo 在 MVP 階段略顯過重，以 npm workspaces 即可。

4.3 Redis Pub/Sub 用於跨 Pod 廣播

理由： Redis 是唯一共享狀態層；pattern subscription 確保新房間自動被訂閱；MVP 為單實例，Pub/Sub 退化為本地呼叫，overhead 可忽略。

取捨： Redis Pub/Sub 為 at-most-once 語意，訊息不持久化；Pod 崩潰後客戶端依賴 WS 重連後的 ROOM_STATE_FULL 補齊狀態。

4.4 不使用 ORM

理由： Room 是聚合根，以 JSON 整包讀取/寫入更符合 Redis 用途；ORM 為 SQL 導向；原子操作需直接控制 Redis 命令。

取捨： 需手動維護 JSON 序列化/反序列化；RoomRepository 單元測試須 mock RedisClient。

4.5 Constructor Injection + container.ts 工廠

理由： 無框架 DI 對 Vitest 測試友善，直接傳入 mock 物件，不需 @injectable() 裝飾器；container.ts 作為組裝根，依賴圖一目了然；避免引入 reflect-metadata polyfill 增加 bundle 大小。

4.6 JWT HS256 (jose) 作為 Session Token

Payload： { playerId, roomCode, role: "host" | "player", exp }。Token 有效期： 6 小時（21600 秒）。

已知安全取捨： JWT exp 僅在 WS Upgrade 握手時驗證；連線建立後不重驗 exp。此為 MVP 接受的安全取捨——遊戲時長通常遠低於 6h。Post-MVP 可在 WsSession 加入定時 token 輪替機制。

4.7 MVP 單實例，HPA 為 Post-MVP

理由： PRD 目標 50 人連線在單實例 Node.js 可達成；避免過早投入 sticky session、Redis Sentinel 等複雜度；架構已預留多 Pod 設計（Pub/Sub、RoomRepository 無本地狀態）。

Redis 失敗降級： ioredis error 事件觸發時，GET /ready 回傳 503，K8s readinessProbe 將 Pod 移出 Service endpoints；ioredis 指數退避重連，恢復後 WS 客戶端重連取得最新 ROOM_STATE_FULL。

6. 狀態管理策略

6.1 Redis 為唯一真相來源

所有房間狀態（Room JSON）以 room:{code} key 儲存於 Redis，Pod 本地記憶體不快取業務狀態。

Redis Key內容TTL
room:{code}Room JSON（含 players、kickedPlayerIds、revealedCount 快照）waiting/running: 24h；finished: 1h
room:{code}:ladderLadderData JSON（seed、seedSource、segments、results）；BEGIN_REVEAL 時才創建同上
room:{code}:revealedCountInteger，INCR 原子遞增同上

seed 安全邊界（公開時機）

房間狀態seed 存在seed 對客戶端可見
waiting / running否（尚未生成）否
revealing是（存於 Redis）否（LadderDataPublic 省略 seed）
finished是是（含於 ROOM_STATE 廣播）

7. 本地計時器所有權

計時器所有者模組觸發條件Pod 重啟後恢復
auto-reveal intervalGameService（靜態 Map roomCode → NodeJS.Timeout）SET_REVEAL_MODE mode=auto；intervalSec 1-30sHost 須重新送 SET_REVEAL_MODE 恢復計時；Client 收到 ROOM_STATE_FULL 且 autoRevealActive=true 時自動重發
host-transfer grace（60s）WsSession（disconnect event 觸發）WS close event（host 斷線）Pod 重啟後 timer 消失；下一個到達事件補跑 hostTransferIfNeeded()
計時器不序列化至 Redis，以保持 Pod 無狀態。Pod 重啟後，auto-reveal 依賴 Host 主動恢復，host-transfer 依賴下一個到達事件補跑。此為 MVP 單 Pod 下可接受的 trade-off。

8. 安全邊界設計

8.1 JWT 驗證層級

  1. WS Upgrade 握手：WsServer.handleUpgrade 驗證 JWT 簽章、exp、roomCode、playerId 一致性
  2. WsMessageHandler：二次驗證 role（host/player）+ room.hostId 比對，防止 JWT role 偽造
  3. HTTP 路由：Fastify preHandler 驗證 Authorization header JWT

8.2 速率限制

  • WS 訊息：WsSession 限制 60 msg/min/連線
  • HTTP POST /rooms：10 req/min/IP
  • HTTP POST /rooms/:code/players：20 req/min/IP
  • 其他 HTTP：100 req/min/IP

8.3 被踢除玩家 WS 攔截

若 playerId 在 kickedPlayerIds 清單中，WS Upgrade 使用 close code 4003（Application-level: Player Kicked）關閉連線，並在關閉前發送 JSON frame { type: "PLAYER_KICKED", ... }，讓前端可區分一般認證失敗（403）與被踢（4003）。
