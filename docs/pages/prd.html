PRD — Ladder Room Online

版本：v1.3   日期：2026-04-19   基於 PDD：v2.2
作者：AI PM Agent   狀態：Draft

執行摘要

Ladder Room Online 是一款基於 HTML5 Canvas 的多人線上爬樓梯抽獎遊戲，支援最多 50 人同房間即時參與。主持人建立房間並掌控揭曉節奏，玩家透過 6 碼房間碼加入並即時觀看路徑動畫；Mulberry32 PRNG 算法確保每局結果在所有客戶端 100% 一致，消除舞弊疑慮，適用於團隊抽獎、課堂點名、活動互動等場景。

使用者故事 — 主持人（Host）

US-H01：建立房間

作為主持人，我想要建立一個帶有唯一 6 碼房間碼的房間，以便玩家能快速找到並加入我的抽獎活動。

  • AC-H01-1：Given 主持人點擊「建立房間」，When 系統接受請求，Then 於 2 秒內回傳 6 碼 Room Code（字元集 [A-HJ-NP-Z2-9]）且在 Redis 中不重複。
  • AC-H01-2：Given 房間建立成功，Then 頁面顯示 Room Code、玩家列表、設定中獎名額輸入框，狀態為 waiting。
  • AC-H01-3：Given 建立房間因網路異常失敗，Then 顯示「建立失敗，請重試」，不產生孤立房間。
  • AC-H01-4：Given Room Code 碰撞超過 10 次，Then 回傳 ROOM_CODE_GENERATION_FAILED，前端顯示「建立失敗，請重試」。

US-H02：設定中獎名額

作為主持人，我想要設定中獎名額（W），以便控制本局有多少玩家會獲獎。

  • AC-H02-1：Given 狀態為 waiting，When 主持人填入合法 W（1 ≤ W ≤ N-1），Then 系統接受並廣播 ROOM_STATE。
  • AC-H02-2：Given W ≤ 0 或 W ≥ N，Then 顯示「中獎名額須介於 1 到玩家數減 1 之間」，不更新狀態。
  • AC-H02-3：Given 再玩一局後 W 越界，Then 自動重設 W 為 null，通知主持人重新設定。

US-H03：開始遊戲

作為主持人，我想要在所有玩家就緒後開始遊戲，以便正式進入揭曉流程。

  • AC-H03-1：Given N ≥ 2 且 W 已設定，When 主持人點擊「開始遊戲」，Then 狀態轉為 running，廣播 ROOM_STATE（含 rowCount）。
  • AC-H03-2：Given W 尚未設定，Then 拒絕並回傳「請先設定中獎名額」，狀態維持 waiting。
  • AC-H03-4：Given N < 2，Then 回傳 INSUFFICIENT_PLAYERS，前端顯示「人數不足（至少需要 2 位玩家）」。
  • AC-H03-5：rowCount 公式為 clamp(N×3, 20, 60)（N=3 → 20；N=10 → 30；N=21 → 60）。

US-H04：手動逐步揭曉

作為主持人，我想要以手動逐步揭曉每條路徑，以便配合現場節奏製造懸念。

  • AC-H04-1：When 主持人點擊「開始揭曉」（BEGIN_REVEAL），Then 伺服器廣播 ROOM_STATE（status=revealing），操作須在 1 秒內完成。
  • AC-H04-2：When 主持人點擊「下一位」，Then 廣播 REVEAL_INDEX，所有客戶端同步播放動畫。
  • AC-H04-3：Given 所有路徑已揭曉，Then「下一位」按鈕 disabled，頁面顯示「結束本局」按鈕。

US-H05：自動揭曉

作為主持人，我想要設定自動揭曉間隔（1～30 秒），以便讓遊戲自動進行。

  • AC-H05-1：Given T 合法（1 ≤ T ≤ 30 整數），Then 每隔 T 秒自動廣播 REVEAL_INDEX。
  • AC-H05-2：Given 自動揭曉進行中切換回手動，Then 計時停止，已揭曉項目不受影響。
  • AC-H05-3：Given T 非整數或超範圍，Then 回傳 INVALID_AUTO_REVEAL_INTERVAL，顯示「間隔須為 1～30 的整數」。

US-H06：一鍵全揭

作為主持人，我想要一鍵揭曉全部路徑，以便在時間緊迫時快速完成抽獎。

  • AC-H06-1：When 主持人點擊「全部揭曉」，Then 廣播 REVEAL_ALL，2 秒內完成渲染；若超時則跳至終止幀。
  • AC-H06-2：REVEAL_ALL 廣播後，伺服器自動觸發 finished 狀態轉換，無需主持人再次確認。

US-H07：踢除玩家

作為主持人，我想要踢除指定玩家，以便移除測試帳號或不當參與者。

  • AC-H07-1：Given 狀態為 waiting，When 踢除玩家，Then 廣播 PLAYER_KICKED，玩家列表即時更新。
  • AC-H07-2：Given 玩家被踢後嘗試以相同 playerId 重連，Then 伺服器拒絕並回傳 PLAYER_KICKED。
  • AC-H07-3：Given 狀態為 running 或之後，Then 拒絕踢除並回傳 INVALID_STATE。
  • AC-H07-5：Given 主持人點擊「再玩一局」，Then kickedPlayerIds 清空，前一局被踢者可加入新局。

US-H08：再玩一局

作為主持人，我想要在一局結束後發起「再玩一局」，以便用同一房間繼續下一輪。

  • AC-H08-1：Given 狀態為 finished，When 主持人點擊「再玩一局」，Then 剔除 isOnline=false 玩家，重置為 waiting，廣播新 ROOM_STATE。
  • AC-H08-3：Given 在線玩家數 < 2，Then 拒絕並回傳 INSUFFICIENT_PLAYERS。

使用者故事 — 玩家（Player）

US-P01：加入房間

  • AC-P01-1：Given 有效暱稱及正確 Room Code，Then 1.5 秒內進入等待畫面，玩家列表顯示自己的暱稱。
  • AC-P01-2：Given 暱稱重複，Then 回傳 NICKNAME_TAKEN，提示「此暱稱已被使用，請換一個」（區分大小寫）。
  • AC-P01-3：Given 空白暱稱或超過 20 字元，Then 前端即時顯示錯誤，送出按鈕保持 disabled。
  • AC-P01-4：Given 房間已達 50 人，Then 回傳 ROOM_FULL，顯示「房間已滿，無法加入」。

US-P02：即時玩家列表

  • AC-P02-1：Given 有新玩家加入，When 伺服器廣播 ROOM_STATE，Then 所有在線玩家 2 秒內看到更新。
  • AC-P02-2：Given 玩家離線，Then 標記 isOnline=false，其他玩家看到「離線」標記，不立即移除。

US-P03：揭曉動畫

  • AC-P03-1：輪到自己時，Canvas 高亮自己的路徑，動畫 FPS 桌機 ≥ 30、手機 ≥ 24。
  • AC-P03-2：其他玩家路徑揭曉時，自己的路徑以灰色呈現，已揭曉結果持久顯示。

US-P04：確認中獎結果

  • AC-P04-1：自己的路徑揭曉完畢，顯示「恭喜中獎！」或「未中獎」，結果與伺服器 payload 一致。

US-P05：斷線重連

  • AC-P05-1：Given localStorage 存有 playerId，系統自動重連，回傳完整房間狀態；重連時間 < 3 秒。
  • AC-P05-2：Given 重連時狀態已為 revealing，直接呈現已揭曉靜態結果，不重播動畫。
  • AC-P05-3：Given localStorage 遺失（Ghost Player），視為新玩家，暱稱重複則提示衝突。

US-P06：被踢通知

  • AC-P06-1：收到 PLAYER_KICKED，立即顯示「你已被主持人移出房間」，WebSocket 關閉，顯示「回首頁」按鈕。
  • AC-P06-2：嘗試以相同 playerId 重連，回傳 PLAYER_KICKED，顯示「你已被移出此房間，無法重新加入」。

功能需求（FR）

FR需求優先級
FR-01房間建立與管理：6 碼 Room Code、TTL 4 小時、碰撞重試最多 10 次P0
FR-02玩家加入與身份管理：UUID playerId、暱稱驗證、50 人上限、isOnline 狀態P0
FR-03樓梯生成算法：Mulberry32 PRNG + djb2 hash + Fisher-Yates 洗牌P0
FR-04房間狀態機：waiting → running → revealing → finished → waitingP0
FR-05揭曉控制：手動 / 自動（1-30s）/ 一鍵全揭P0
FR-06WebSocket 事件規格：7 種伺服器事件，11 種客戶端操作P0
FR-07斷線重連：playerId 驗證、Ghost Player、Session 替換P1
FR-08再玩一局：移除離線玩家、重置欄位、W 越界重設P1
FR-09踢除玩家：kickedPlayerIds 持久化、同局有效、再玩一局清空P1
FR-10Canvas 動畫渲染：響應式、逐格動畫、requestAnimationFrameP1
FR-11房間碼輸入體驗：自動轉大寫、URL 參數預填P2
FR-12得獎名單展示：完整排序名單、中獎/未中獎標記P2

非功能需求（NFR）

NFR類別目標
NFR-01效能Canvas FPS 桌機 ≥ 30、手機 ≥ 24；WebSocket 延遲 < 2 秒（P95）
NFR-02並發容量100 並發房間 × 50 人 = 5,000 WS 連線；成功率 > 99.5%
NFR-03結果一致性100% 局次所有客戶端結果與伺服器一致，0 容忍差異
NFR-04可靠性重連狀態恢復率 100%，< 3 秒（P95）
NFR-05安全性JWT 驗證、seed 在 finished 前禁止洩漏、WS 訊息上限 64KB
NFR-06前端效能預算JS bundle < 150KB gzip；FCP < 1.5s；LCP < 2.5s；CLS < 0.1
NFR-07測試覆蓋率後端 ≥ 80%；前端 ≥ 70%；E2E 覆蓋所有 P0 Happy Path
NFR-08瀏覽器相容性Chrome 110+、Firefox 110+、Safari 15+；最小 320px 寬度
