📝 PRD — Ladder Room Online

版本：v1.3 ｜ 日期：2026-04-19 ｜ 基於 PDD：v2.2 ｜ 狀態：Draft

執行摘要

Ladder Room Online 是一款基於 HTML5 Canvas 的多人線上爬樓梯抽獎遊戲，支援最多 50 人同房間即時參與。主持人建立房間並掌控揭曉節奏，玩家透過 6 碼房間碼加入並即時觀看路徑動畫；Mulberry32 PRNG 算法確保每局結果在所有客戶端 100% 一致，消除舞弊疑慮，適用於團隊抽獎、課堂點名、活動互動等場景。

使用者故事

主持人（Host）

US-H01：作為主持人，我想要建立一個帶有唯一 6 碼房間碼的房間，以便玩家能快速找到並加入我的抽獎活動。

  • AC-H01-1：Given 主持人點擊「建立房間」，When 系統接受請求，Then 於 2 秒內回傳 6 碼 Room Code（字元集 ABCDEFGHJKLMNPQRSTUVWXYZ23456789）且在 Redis 中不重複。
  • AC-H01-2：Given 房間建立成功，When 主持人進入房間，Then 頁面顯示 Room Code、目前玩家列表（空）、設定中獎名額的輸入框，狀態為 waiting。
  • AC-H01-3：Given 建立房間的請求因網路異常失敗，When 系統偵測到錯誤，Then 顯示「建立失敗，請重試」提示並保留輸入內容，不產生孤立房間。
  • AC-H01-4：Given Room Code 生成碰撞次數超過 10 次，Then 回傳 ROOM_CODE_GENERATION_FAILED 錯誤。

US-H02：作為主持人，我想要設定中獎名額（W），以便控制本局有多少玩家會獲獎。

  • AC-H02-1：Given 房間狀態為 waiting，When 主持人填入 W（1 ≤ W ≤ N-1），Then 系統接受並廣播更新後的 ROOM_STATE。
  • AC-H02-2：Given 主持人輸入 W ≤ 0 或 W ≥ N，Then 系統顯示「中獎名額須介於 1 到玩家數減 1 之間」錯誤。
  • AC-H02-3：Given 再玩一局且剔除離線玩家後 W ≥ 新 N，Then 自動將 W 重設為 null 並通知主持人。

US-H03：作為主持人，我想要在所有玩家就緒後開始遊戲，以便正式進入揭曉流程。

  • AC-H03-1：Given 房間狀態為 waiting 且 N ≥ 2 且 W 已設定，When 主持人點擊「開始遊戲」，Then 狀態轉為 running，seed 生成與樓梯計算原子完成，廣播 ROOM_STATE（status=running，含 rowCount）；seed 及完整樓梯資料在 status=finished 前禁止傳送給任何客戶端。
  • AC-H03-4：Given 房間只有 Host 一人（N < 2），Then 系統拒絕並回傳 INSUFFICIENT_PLAYERS 錯誤。
  • AC-H03-5：rowCount 公式為 clamp(N×3, 20, 60)。

US-H04：作為主持人，我想要以手動逐步揭曉每條路徑，以便配合現場節奏製造懸念。

  • AC-H04-1：Given 房間狀態為 running，When 主持人點擊「開始揭曉」，Then 伺服器執行狀態轉換（running → revealing）並廣播 ROOM_STATE，操作須在 1 秒內完成。
  • AC-H04-2：Given 狀態為 revealing，When 主持人點擊「下一位」，Then 伺服器廣播 REVEAL_INDEX，所有客戶端同步播放行走動畫。
  • AC-H04-4：Given 所有路徑已揭曉且主持人點擊「結束本局」，Then 狀態轉為 finished，廣播含完整得獎名單及 seed 的 ROOM_STATE。

US-H05：作為主持人，我想要設定自動揭曉間隔（1～30 秒/格）。

  • AC-H05-1：Given 主持人選擇自動模式並設定間隔秒數 T（1 ≤ T ≤ 30，T 須為正整數），When 遊戲進入 revealing，Then 每隔 T 秒伺服器自動廣播下一個 REVEAL_INDEX。
  • AC-H05-3：Given 主持人輸入非法 T，Then 回傳 INVALID_AUTO_REVEAL_INTERVAL 錯誤。

US-H06：作為主持人，我想要一鍵揭曉全部路徑。

  • AC-H06-1：Given 狀態為 revealing 且尚有未揭曉路徑，When 主持人點擊「全部揭曉」，Then 伺服器廣播 REVEAL_ALL，所有客戶端 2 秒內完成渲染；超時則跳至終止幀。

US-H07：作為主持人，我想要踢除指定玩家。

  • AC-H07-1：Given 房間狀態為 waiting，When 主持人對某玩家點擊「踢除」，Then 伺服器廣播 PLAYER_KICKED，被踢玩家跳轉至「你已被移出房間」頁面。
  • AC-H07-3：Given 房間狀態為 running 或之後，When 主持人嘗試踢除玩家，Then 系統拒絕並回傳 INVALID_STATE。

US-H08：作為主持人，我想要在一局結束後發起「再玩一局」。

  • AC-H08-1：Given 狀態為 finished，When 主持人點擊「再玩一局」，Then 系統剔除 isOnline=false 的玩家，重置房間狀態為 waiting。

玩家（Player）

US-P01：作為玩家，我想要輸入暱稱並透過 6 碼房間碼加入房間。

  • AC-P01-1：Given 玩家輸入有效暱稱（1～20 個 Unicode 字元）及正確 Room Code，Then 從送出按鈕點擊起計 1.5 秒內進入等待畫面。
  • AC-P01-2：Given 暱稱重複（區分大小寫），Then 回傳 NICKNAME_TAKEN 錯誤。
  • AC-P01-4：Given 房間人數已達 50 人上限，Then 伺服器回傳 ROOM_FULL 錯誤。

US-P03：作為玩家，我想要觀看自己路徑的揭曉動畫。

  • AC-P03-1：Canvas 高亮自己的路徑，動畫 FPS 桌機 ≥ 30、手機 ≥ 24。

US-P05：作為玩家，我想要在斷線後能重新連上房間並恢復狀態。

  • AC-P05-1：Given 玩家斷線後重新開啟頁面且 localStorage 中存有 playerId，Then 系統自動重連，重連時間 < 3 秒。

功能需求

FR-01 房間建立與管理 [P0]

  • FR-01-1：系統須在 2 秒內生成唯一 6 碼 Room Code，字元集：ABCDEFGHJKLMNPQRSTUVWXYZ23456789（排除 O、I、0、1）。
  • FR-01-2：Room Code 在 Redis 中以 TTL = 4 小時儲存；碰撞則重試最多 10 次。
  • FR-01-4：主持人建立房間後取得專屬 token（JWT），所有 Host 操作須附帶此 token 驗證。

FR-03 樓梯生成算法 [P0]

  • FR-03-1：樓梯結構以 Mulberry32 PRNG 生成，seed 以 djb2 hash 將 seed 字串（UUID hex）轉為 uint32 初始化。
  • FR-03-2：橫槓生成規則：每 row 目標 max(1, round(N/4)) 條橫槓；只連接相鄰欄位；同 row 橫槓不重疊。
  • FR-03-3：結果槽指派須滿足 bijection（N 個起點對應 N 個唯一終點），以 Fisher-Yates 洗牌指派 W 個中獎槽位。

FR-04 房間狀態機 [P0]

房間狀態流轉：waiting → running → revealing → finished → waiting（再玩一局）。

FR-06 WebSocket 事件規格 [P0]

伺服器 → 客戶端：ROOM_STATE、ROOM_STATE_FULL、REVEAL_INDEX、REVEAL_ALL、PLAYER_KICKED、SESSION_REPLACED、ERROR

客戶端 → 伺服器：START_GAME、UPDATE_WINNER_COUNT、BEGIN_REVEAL、REVEAL_NEXT、SET_AUTO_REVEAL、REVEAL_ALL、END_GAME、PLAY_AGAIN、KICK_PLAYER、JOIN_ROOM、RECONNECT

非功能需求

不在範圍（Out of Scope）

  1. 使用者帳號系統：MVP 不實作登入、註冊、歷史紀錄查詢。
  2. 行動原生 App：MVP 僅支援行動瀏覽器（Mobile Web）。
  3. 觀眾模式（Spectator）：MVP 不支援純觀看不參與的角色。
  4. 多房間管理後台：MVP 不提供管理員儀表板。
  5. 自訂獎品名稱或分組抽獎：MVP 結果槽僅區分「中獎」與「未中獎」。
  6. 動畫主題或皮膚自訂：MVP 使用單一視覺主題。
  7. 抽獎結果匯出（PDF / CSV）：MVP 不提供結果下載功能。
  8. 國際化（i18n）多語言支援：MVP 以繁體中文為唯一介面語言。
  9. 伺服器端水平擴展：MVP 以單一 Node.js 實例部署。
  10. 聲音效果與背景音樂：MVP 不包含音效。
