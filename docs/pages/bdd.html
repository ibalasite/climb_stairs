Feature: 遊戲流程控制（game-flow.feature）

Feature: 遊戲流程控制
  As a 主持人
  I want to 設定中獎名額並在玩家就緒後開始遊戲
  So that 伺服器生成確定性樓梯結構並讓所有客戶端同步進入揭曉流程

  Background:
    Given 房間碼 "BETA23" 已存在且狀態為 "waiting"
    And 主持人已透過 hostToken 驗證身份

  @AC-GAME-WIN-001 @P0
  Scenario: 主持人設定合法中獎名額
    Given 房間有 5 名玩家（N=5，含主持人）
    When 主持人設定中獎名額 W=2（1 <= W <= N-1=4）
    Then 伺服器接受設定
    And 所有在線玩家收到廣播的 ROOM_STATE
    And ROOM_STATE payload 中 winnerCount 為 2

  @AC-GAME-WIN-002 @P0
  Scenario Outline: 主持人設定非法中獎名額時收到 INVALID_PRIZES_COUNT 錯誤
    Given 房間有 <N> 名玩家（含主持人）
    When 主持人設定中獎名額 W=<W>
    Then 伺服器回傳錯誤碼 "INVALID_PRIZES_COUNT"
    And 房間 winnerCount 維持不變

    Examples:
      | N  | W  | 說明              |
      | 5  | 0  | W=0，低於下限      |
      | 5  | -1 | W<0，負數         |
      | 5  | 5  | W=N，等於上限違規  |
      | 5  | 6  | W>N，超過玩家數   |

  @AC-GAME-START-001 @P0
  Scenario: 主持人滿足條件後成功開始遊戲
    Given 房間有 3 名玩家（N=3）且 winnerCount=1 已設定
    When 主持人點擊「開始遊戲」
    Then HTTP 回應狀態碼為 200
    And 伺服器廣播 ROOM_STATE，status 為 "running"
    And 所有客戶端收到完全一致的 ladderMap

  @AC-GAME-START-002 @P0
  Scenario: W 尚未設定時主持人開始遊戲被拒絕
    Given 房間有 3 名玩家且 winnerCount 尚未設定（為 null）
    When 主持人點擊「開始遊戲」
    Then 伺服器回傳錯誤碼 "PRIZES_NOT_SET"
    And 房間狀態維持 "waiting"

  @AC-GAME-START-003 @P0
  Scenario: 房間只有主持人一人時開始遊戲被拒絕
    Given 房間只有主持人一人（N=1）
    When 主持人點擊「開始遊戲」
    Then 伺服器回傳錯誤碼 "INSUFFICIENT_PLAYERS"
    And 前端顯示「人數不足（至少需要 2 位玩家）」
    And 房間狀態維持 "waiting"

  @AC-GAME-START-004 @P0
  Scenario Outline: 遊戲開始後 rowCount 符合 clamp(N*3, 20, 60) 公式
    Given 房間有 <N> 名玩家且 winnerCount 已合法設定
    When 主持人成功開始遊戲
    Then 伺服器廣播的 ROOM_STATE 中 ladder.rowCount 為 <expected_rowCount>

    Examples:
      | N  | expected_rowCount |
      | 2  | 20                |
      | 3  | 20                |
      | 10 | 30                |
      | 20 | 60                |
      | 50 | 60                |

Feature: 路徑揭曉流程（reveal-flow.feature）

Feature: 路徑揭曉流程
  As a 主持人
  I want to 手動逐步、自動定時或一鍵揭曉所有玩家的路徑
  So that 能配合現場節奏製造懸念

  Background:
    Given 房間碼 "GAMMA4" 存在且狀態為 "revealing"
    And 房間有 4 名玩家（N=4），winnerCount=1
    And 主持人已驗證身份
    And 尚有未揭曉的路徑

  @AC-REVEAL-001 @P0
  Scenario: 主持人點擊「下一位」後所有客戶端同步播放揭曉動畫
    Given 目前已揭曉 0 條路徑（revealedCount=0）
    When 主持人送出 WS 訊息 REVEAL_NEXT
    Then 伺服器廣播 REVEAL_INDEX 事件給房間內所有連線客戶端
    And REVEAL_INDEX payload 包含 playerIndex、result 及 revealedCount=1

  @AC-REVEAL-002 @P0
  Scenario: 所有路徑揭曉完畢後再點擊「下一位」系統不回應
    Given 所有 4 條路徑已全數揭曉（revealedCount=4）
    When 主持人送出 WS 訊息 REVEAL_NEXT
    Then 伺服器不廣播任何 REVEAL_INDEX 事件
    And 伺服器回傳 ERROR 事件，code 為 "INVALID_STATE"

  @AC-REVEAL-003 @P1
  Scenario: 主持人設定自動揭曉間隔後系統定時廣播
    Given 目前已揭曉 0 條路徑
    When 主持人送出 SET_REVEAL_MODE（mode="auto"，intervalSec=3）
    Then 每隔 3 秒伺服器自動廣播下一個 REVEAL_INDEX
    And 重複直到所有 4 條路徑全部揭曉

  @AC-REVEAL-004 @P1
  Scenario: 自動揭曉進行中切換回手動模式後計時停止
    Given 自動揭曉模式進行中（intervalSec=5），已揭曉 2 條路徑
    When 主持人送出 SET_REVEAL_MODE（mode="manual"）
    Then 伺服器停止自動計時，不再自動廣播 REVEAL_INDEX
    And 已揭曉的 2 條路徑結果不受影響

  @AC-REVEAL-005 @P0
  Scenario: 主持人點擊「全部揭曉」後伺服器廣播 REVEAL_ALL
    Given 目前已揭曉 1 條路徑，剩餘 3 條未揭曉
    When 主持人送出 WS 訊息 REVEAL_ALL_TRIGGER
    Then 伺服器廣播 REVEAL_ALL 事件給所有客戶端
    And REVEAL_ALL payload 包含所有剩餘 3 條路徑的 ResultSlot 資料

  @AC-BEGIN-REVEAL-001 @P0
  Scenario: 主持人在 running 狀態觸發 BEGIN_REVEAL 後房間進入 revealing 狀態
    Given 房間碼 "GAMMA4" 存在且狀態為 "running"
    When 主持人送出 WS 訊息 BEGIN_REVEAL
    Then 伺服器廣播 ROOM_STATE，status 為 "revealing"
    And revealedCount 為 0

Feature: 玩家管理（player-management.feature）

Feature: 玩家管理
  As a 主持人
  I want to 管理等待大廳中的玩家
  So that 確保房間內玩家名單符合預期

  @AC-KICK-001 @P0
  Scenario: 主持人踢除在線玩家，被踢玩家收到通知並導向首頁
    Given 等待大廳有玩家 "Bob"（isOnline=true）
    When 主持人踢除 "Bob"
    Then "Bob" 的客戶端收到 PLAYER_KICKED 事件
    And "Bob" 的 WS 連線立即被 Server 關閉（close code 4003）
    And "Bob" 客戶端清除 localStorage 中的 playerId
    And "Bob" 客戶端 3 秒後導向首頁
    And 其他玩家收到 ROOM_STATE（"Bob" 不在 players 陣列）

  @AC-KICK-002 @P0
  Scenario: 被踢玩家嘗試重連，Server 回傳 PLAYER_KICKED 錯誤
    Given "Bob" 已被踢除（playerId 在 kickedPlayerIds）
    When "Bob" 以原 playerId 嘗試建立 WS 連線
    Then Server 回傳 close code 4003
    And 先發送 JSON frame { type: "PLAYER_KICKED", ... }

  @AC-PLAY-AGAIN-001 @P0
  Scenario: 主持人點擊「再玩一局」，離線玩家被移除，禁令清除
    Given 房間狀態為 "finished"，有 2 名在線玩家、1 名離線玩家
    And "Bob" 在前一局被踢除（在 kickedPlayerIds）
    When 主持人點擊「再玩一局」
    Then 離線玩家從 players 陣列移除
    And kickedPlayerIds 清空（"Bob" 的踢除禁令解除）
    And 房間狀態重置為 "waiting"
    And Server 廣播 ROOM_STATE（status="waiting"）

  @AC-PLAY-AGAIN-002 @P0
  Scenario: 再玩一局後，前一局被踢玩家可使用新暱稱重新加入
    Given 主持人已點擊「再玩一局」，kickedPlayerIds 已清空
    When 前一局被踢除的玩家以新暱稱 "Bob_New" 嘗試加入
    Then Server 接受加入請求
    And "Bob_New" 成為房間的正式玩家

Feature: 重連處理（reconnect.feature）

Feature: 斷線與重連處理
  As a 玩家或主持人
  I want to 在網路短暫中斷後自動重連並恢復遊戲狀態
  So that 不因短暫斷線而失去遊戲進度

  @AC-RECONNECT-001 @P0
  Scenario: 玩家帶有效 playerId 重連，Server unicast ROOM_STATE_FULL
    Given 玩家 "Carol" 在 "revealing" 狀態斷線
    And 已揭曉 2 條路徑
    When "Carol" 以原 playerId 重新建立 WS 連線
    Then Server unicast ROOM_STATE_FULL 給 "Carol"
    And ROOM_STATE_FULL 包含完整 players、ladder（LadderDataPublic）、results（已揭曉部分）
    And "Carol" 的 isOnline 更新為 true
    And Server 廣播 ROOM_STATE 至其他玩家（"Carol" isOnline=true）

  @AC-RECONNECT-002 @P0
  Scenario: Host 斷線超過 60 秒，主持權自動移交給下一位在線玩家
    Given 主持人 "Alice" 斷線
    And 房間有另一位在線玩家 "Bob"
    When 60 秒過去，"Alice" 未重連
    Then Server 廣播 HOST_TRANSFERRED 事件
    And ROOM_STATE 廣播中 hostId 更新為 "Bob" 的 playerId

  @AC-RECONNECT-003 @P0
  Scenario: 同一 playerId 從新分頁登入，舊連線收到 SESSION_REPLACED
    Given 玩家 "Dave" 已有活躍 WS 連線（舊分頁）
    When "Dave" 從另一分頁以相同 playerId 建立 WS 連線
    Then 舊連線收到 SESSION_REPLACED 事件
    And 舊連線的 WS 斷線
    And 新連線成為 "Dave" 的活躍 session

Feature: PRNG 確定性驗證（prng.feature）

Feature: PRNG 確定性驗證
  As a 系統
  I want to 確保相同 seed 在前後端產生完全相同的梯子結構
  So that 玩家可在 finished 後自行驗證抽獎公平性（PRD AC-SYS-01）

  @AC-PRNG-001 @P0
  Scenario: 相同 seed 在 packages/shared 前後端共用，產生 100% 相同的梯子結構
    Given 已知 seed = 3141592653，rowCount = 20，colCount = 3
    When 後端（Node.js）以此 seed 執行 generateLadder
    And 前端（Browser）以此 seed 執行 generateLadder
    Then 兩端的 segments 陣列完全相同（長度、每個元素的 row/col 一致）
    And 兩端的 ResultSlots 陣列完全相同（startCol、endCol、isWinner 一致）

  @AC-PRNG-002 @P0
  Scenario: 不同 seed 產生不同梯子結構
    Given seed A = 100000001，seed B = 100000002，其餘參數相同
    When 分別執行 generateLadder
    Then 兩次生成的 segments 陣列不完全相同（至少有一個元素不同）

Feature: Canvas 渲染行為（client-canvas.feature）

Feature: 梯子 Canvas 渲染行為
  As a 玩家
  I want to 在各個遊戲階段看到符合狀態的 Canvas 畫面
  So that 我能清楚掌握遊戲進行狀況並正確辨識自己的結果

  @canvas @waiting @P1
  Scenario: 房間狀態為 waiting 時 Canvas 顯示佔位畫面
    Given 房間狀態為 "waiting"，尚未生成梯子資料
    When Canvas 完成初次渲染
    Then Canvas 顯示灰色佔位軌道，對應每位已加入的玩家
    And Canvas 不顯示任何橫槓或路徑

  @canvas @running @P0
  Scenario: 房間狀態切換為 running 時 Canvas 渲染完整梯子結構
    Given 房間狀態從 "waiting" 切換為 "running"
    And 伺服器廣播包含 ladderData 的 ROOM_STATE
    When Canvas 接收到新的梯子資料
    Then Canvas 顯示所有垂直軌道，數量等於玩家人數
    And Canvas 顯示所有水平橫槓，位置與伺服器資料一致

  @canvas @revealing @P0
  Scenario: 觸發揭曉時 Canvas 播放路徑動畫
    Given 房間狀態為 "running"，梯子已渲染
    When 主持人觸發揭曉，Canvas 接收到 REVEAL_INDEX 事件
    Then Canvas 沿對應玩家的行進路徑播放移動動畫
    And 動畫完成後路徑保持彩色高亮靜止顯示

  @canvas @finished @P0
  Scenario: 所有路徑揭曉完成後 Canvas 顯示最終結果標籤
    Given 所有玩家的路徑均已揭曉
    When Canvas 完成最後一條路徑的渲染
    Then Canvas 底部每條軌道終點顯示「中獎」或「未中獎」標籤
    And 中獎標籤以金色顯示，未中獎標籤以灰色顯示

Feature: 頁面流程與視圖切換（client-navigation.feature）

Feature: 頁面流程與視圖切換
  As a 玩家或主持人
  I want to 在不同遊戲階段被引導到對應的頁面
  So that 我能看到符合當前狀態的 UI

  @navigation @lobby @P0
  Scenario: 首次進入應用程式時顯示首頁
    When 頁面載入完成
    Then 顯示「建立房間」與「加入房間」兩個操作入口
    And 不顯示梯子 Canvas、玩家列表或結果列表

  @navigation @lobby @P0
  Scenario: 主持人填寫暱稱後建立房間，進入等待大廳
    Given 使用者在首頁輸入暱稱 "Alice"
    When 使用者點擊「建立房間」，伺服器回傳房間碼 "EPS6"
    Then 畫面切換至等待大廳視圖
    And 等待大廳顯示房間碼 "EPS6"
    And 玩家列表中顯示 "Alice"（主持人標記）

  @navigation @lobby @P0
  Scenario: 玩家輸入房間碼與暱稱後加入房間，進入等待大廳
    Given 使用者在首頁輸入暱稱 "Bob" 與房間碼 "EPS6"
    When 使用者點擊「加入房間」
    Then 畫面切換至等待大廳視圖
    And 玩家列表中顯示 "Bob"

  @navigation @waiting @P0
  Scenario: 主持人點擊「開始遊戲」後所有人切換至遊戲頁面
    Given 等待大廳有 3 名玩家且中獎名額已設定
    When 主持人點擊「開始遊戲」
    And 伺服器廣播 ROOM_STATE（status="running"）
    Then 主持人畫面切換至遊戲頁面
    And 所有玩家畫面同步切換至遊戲頁面

Feature: 錯誤狀態 UI 回饋（client-error.feature）

Feature: 錯誤狀態的 UI 回饋行為
  As a 玩家或主持人
  I want to 在操作失敗時立刻看到清楚的錯誤提示
  So that 我能理解發生了什麼問題並採取正確的後續行動

  @error @join @P0
  Scenario: 暱稱為空時送出加入請求，顯示 inline 錯誤提示
    Given 我在首頁的加入房間表單
    When 我未填寫暱稱直接點擊「加入房間」
    Then 暱稱欄位旁顯示「請輸入暱稱」錯誤提示
    And 不送出任何 HTTP 請求

  @error @join @P0
  Scenario: 房間不存在時顯示 Toast 錯誤訊息
    Given 我在首頁輸入暱稱 "Bob" 與房間碼 "XXXX"
    When 我點擊「加入房間」，伺服器回傳錯誤碼 "ROOM_NOT_FOUND"
    Then 頁面顯示 error Toast「找不到此房間」
    And 頁面維持在首頁視圖

  @error @join @P0
  Scenario: 暱稱已被使用時顯示 Toast 錯誤訊息
    Given 房間 "EPS6" 中已有暱稱為 "Carol" 的玩家
    When 我以暱稱 "Carol" 嘗試加入房間 "EPS6"
    And 伺服器回傳錯誤碼 "NICKNAME_TAKEN"
    Then 頁面顯示 error Toast「此暱稱已被使用，請換一個」

  @error @host @P0
  Scenario: 人數不足時主持人開始遊戲，顯示 Toast 錯誤
    Given 等待大廳只有主持人一人
    When 主持人點擊「開始遊戲」
    And 伺服器回傳錯誤碼 "INSUFFICIENT_PLAYERS"
    Then 頁面顯示 error Toast「人數不足（至少需要 2 位玩家）」
    And 房間視圖維持在等待大廳

  @error @kicked @P0
  Scenario: 被主持人踢出房間時顯示提示並返回首頁
    Given 我在房間 "EPS6" 的等待大廳
    When 主持人將我踢出，伺服器發送 PLAYER_KICKED 事件
    Then 頁面顯示 error Toast「你已被踢出房間」
    And 畫面切換至首頁（lobby 視圖）
    And localStorage 中的 token 被清除

  @error @toast @P1
  Scenario: Toast 錯誤訊息在 3.5 秒後自動消失
    Given 頁面顯示一條 error Toast
    When 3.5 秒後
    Then Toast 自動淡出並從畫面移除

Feature: 房間生命週期（room-lifecycle.feature）

Feature: 房間生命週期
  As a 系統
  I want to 正確管理房間的建立、狀態轉換與過期清除
  So that 確保資源使用合理且不留殘餘狀態

  @lifecycle @P0
  Scenario: 建立房間後 Redis 存在對應的 key，TTL 為 24h
    When POST /api/rooms 成功
    Then Redis key "room:{code}" 存在
    And TTL 介於 86399 到 86400 秒之間（24h）

  @lifecycle @P0
  Scenario: waiting 狀態房間 4 小時無活動後自動清除
    Given 房間建立後無任何玩家活動
    When 4 小時後
    Then Redis key "room:{code}" 不存在（TTL 到期）

  @lifecycle @P0
  Scenario: finished 狀態房間 TTL 縮短至 1 小時
    Given 房間狀態轉為 "finished"
    When END_GAME 觸發
    Then Redis 為 "room:{code}" 設定 EXPIRE 3600（1h）

  @lifecycle @P0
  Scenario: 所有玩家斷線後 5 分鐘房間自動清除
    Given 房間有 2 名玩家，均在線
    When 所有玩家斷線，最後一個 WS 連線關閉
    Then Redis 為 "room:{code}" 設定 EXPIRE 300（5min）
    And 5 分鐘後房間消失

測試執行說明

E2E 測試（Playwright）

# 安裝 Playwright
npm install -D @playwright/test
npx playwright install

# 執行所有 E2E 測試
npx playwright test

# 執行特定 feature（需有 Cucumber / Playwright 橋接）
npx playwright test --grep "遊戲流程"

Unit / Integration 測試（Vitest）
