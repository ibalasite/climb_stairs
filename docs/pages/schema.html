SCHEMA — Redis Schema Design

版本：v1.1  |  日期：2026-04-19  |  基於 EDD v1.3

Redis 是唯一的持久層。不使用 SQL 資料庫。所有狀態以 Redis key-value 結構儲存，並設定明確的 TTL。

1. 資料儲存概覽

持久層選擇

Ladder Room Online 使用 Redis 作為唯一的持久層：

  • 房間生命週期短（數分鐘到數小時），天然適合 TTL 驅動的資料管理
  • WebSocket 廣播需要低延遲讀寫，Redis 的亞毫秒響應完全符合
  • 原子操作（INCR、SETNX、MULTI/EXEC、WATCH）消除競態條件
  • 記憶體用量可預測（每房間 ~88 KB，100 房間僅 ~8.8 MB）

儲存的資料類型

類別儲存內容
房間狀態status、hostId、winnerCount（number | null）、revealMode、autoRevealIntervalSec、createdAt、updatedAt
玩家狀態players 陣列（nickname、colorIndex、isHost、isOnline）
梯子資料seed（number）、seedSource（UUID hex）、rowCount、colCount、segments 陣列
揭示計數revealedCount 原子計數器
踢除名單kickedPlayerIds Set（同時冗余儲存於 Room JSON）
Session 對應playerId → sessionId Hash

TTL 策略

房間狀態TTL觸發條件
waiting / running / revealing24h建立或任何狀態變更時重設
finished1h轉換至 finished 狀態時設定
所有玩家離線5min (300s)最後一個 WS 連線關閉時 EXPIRE
Pod 重啟維持現有 TTL無連線的房間不重設，自然過期

2. Redis Key Schema

完整鍵清單

Key 模式TypeValueTTL用途
room:{code}String (JSON)序列化的 Room 物件（含 players 陣列）24h（每次狀態變更重設）房間主狀態
room:{code}:ladderString (JSON)LadderData JSON（seed、segments、results）與主鍵同步梯子結構與結果（BEGIN_REVEAL 時才創建）
room:{code}:revealedCountString (counter)整數字串，如 "0" 到 "N"與主鍵同步揭示進度原子計數器
room:{code}:kickedSet{ playerId1, playerId2, ... }與主鍵同步踢除玩家 ID 集合，用於重連檢查
room:{code}:sessionsHash{ playerId: sessionId }與主鍵同步追蹤活躍 WebSocket Session

範例鍵

room:ALPHA1
room:ALPHA1:ladder
room:ALPHA1:revealedCount
room:ALPHA1:kicked
room:ALPHA1:sessions

3. Room 物件 JSON 結構

Stage 1: waiting（剛建立，等待玩家加入）

{
  "code": "ALPHA1",
  "status": "waiting",
  "hostId": "player-uuid-001",
  "winnerCount": null,
  "revealMode": "manual",
  "autoRevealIntervalSec": null,
  "kickedPlayerIds": [],
  "createdAt": 1745049600000,
  "updatedAt": 1745049600000,
  "players": [
    {
      "id": "player-uuid-001",
      "nickname": "Alice",
      "colorIndex": 0,
      "isHost": true,
      "isOnline": true,
      "joinedAt": 1745049600000
    }
  ]
}

room:ALPHA1:ladder → 空字串或鍵不存在

room:ALPHA1:revealedCount → "0"

Stage 2: running（START_GAME 已觸發，seedSource 已生成，梯子尚未生成）

重要：START_GAME 只生成 seedSource（UUID v4）並計算 rowCount，不生成梯子結構。梯子（LadderData、ResultSlots）延遲至 BEGIN_REVEAL 時原子生成。room:{code}:ladder 鍵在此階段不存在。

{
  "code": "ALPHA1",
  "status": "running",
  "hostId": "player-uuid-001",
  "winnerCount": 1,
  "kickedPlayerIds": ["player-uuid-004"],
  "players": [
    { "id": "player-uuid-001", "nickname": "Alice", "isHost": true, "isOnline": true },
    { "id": "player-uuid-002", "nickname": "Bob",   "isHost": false, "isOnline": true },
    { "id": "player-uuid-003", "nickname": "Carol", "isHost": false, "isOnline": false }
  ]
}

room:ALPHA1:ladder → 不存在（梯子尚未生成；BEGIN_REVEAL 時才創建）

Stage 3: revealing（BEGIN_REVEAL 已觸發梯子生成）

房間主鍵 JSON 同 Stage 2，但 status 更新為 "revealing"。

room:ALPHA1:ladder → BEGIN_REVEAL 時原子生成，此時已存在（含 seed、seedSource、segments、results）。

seed 存在於 Redis，但 ROOM_STATE_FULL 以 LadderDataPublic（省略 seed/seedSource）傳送客戶端，防止預測抽獎結果。

Stage 4: finished（全部揭示完畢）

此時 TTL 降為 1h，room:ALPHA1:revealedCount → "3"（等於 players.length）。

此階段 ROOM_STATE 廣播首次對客戶端公開 seed 與 seedSource（可審計性需求 PRD FR-03-1）。

4. 資料模型（TypeScript → Redis 對應）

interface Room {
  readonly code: string;
  status: 'waiting' | 'running' | 'revealing' | 'finished';
  hostId: string;
  winnerCount: number | null;         // null until host sets it
  revealMode: 'manual' | 'auto';
  autoRevealIntervalSec: number | null; // 1-30s; null in manual mode
  kickedPlayerIds: readonly string[];
  createdAt: number;  // Unix ms
  updatedAt: number;
  readonly players: readonly Player[];
}

interface Player {
  readonly id: string;       // UUID v4
  nickname: string;
  colorIndex: number;
  isHost: boolean;
  isOnline: boolean;
  joinedAt: number;          // Unix ms
}

// 完整梯子資料（儲存於 Redis，含 seed/seedSource）
interface LadderData {
  readonly seed: number;          // Mulberry32 seed = djb2(seedSource) >>> 0
  readonly seedSource: string;    // UUID v4 hex generated at START_GAME
  readonly rowCount: number;      // clamp(N*3, 20, 60)
  readonly colCount: number;      // = N
  readonly segments: readonly LadderSegment[];
  readonly results: readonly ResultSlot[];  // 在 BEGIN_REVEAL 時原子生成
}

// 發送給客戶端的公開版本（省略 seed/seedSource，在 revealing 狀態使用）
interface LadderDataPublic {
  readonly rowCount: number;
  readonly colCount: number;
  readonly segments: readonly LadderSegment[];
}

interface LadderSegment {
  readonly row: number;
  readonly col: number;  // col <-> col+1 has a rung
}

interface PathStep {
  readonly row: number;
  readonly col: number;
  readonly direction: 'down' | 'left' | 'right';
}

interface ResultSlot {
  readonly playerIndex: number;
  readonly playerId: string;
  readonly startCol: number;
  readonly endCol: number;
  readonly isWinner: boolean;
  readonly path: readonly PathStep[];
}

5. 原子操作策略

START_GAME 原子操作

# 使用 WATCH + MULTI/EXEC 確保原子性
WATCH room:{code}
MULTI
  SET room:{code} {updated_room_json_with_status_running}
  EXPIRE room:{code} 86400  # 24h
EXEC

BEGIN_REVEAL Lua Script（原子生成 LadderData + 狀態轉換）

# Lua Script 確保 GenerateLadder 結果與 status=revealing 原子寫入
local room = redis.call('GET', KEYS[1])
-- 解析、生成梯子、計算 results...
redis.call('SET', KEYS[1], new_room_json)         -- status = revealing
redis.call('SET', KEYS[2], ladder_json)           -- room:{code}:ladder
redis.call('EXPIRE', KEYS[1], 86400)
redis.call('EXPIRE', KEYS[2], 86400)

REVEAL_NEXT 原子計數

INCR room:{code}:revealedCount  # 原子遞增，返回新計數
