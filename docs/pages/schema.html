Schema — Redis 資料模型

Redis is the sole persistence layer. There is no SQL database. All state lives in Redis key-value structures with explicit TTLs.

1. 資料儲存概覽

持久層選擇理由

  • 房間生命週期短（數分鐘到數小時），天然適合 TTL 驅動的資料管理
  • WebSocket 廣播需要低延遲讀寫，Redis 的亞毫秒響應完全符合
  • 原子操作（INCR、SETNX、MULTI/EXEC、WATCH）消除競態條件
  • 記憶體用量可預測（每房間 ~88 KB，100 房間僅 ~8.8 MB）

儲存的資料類型

類別儲存內容
房間狀態status、hostId、winnerCount（number | null）、revealMode、autoRevealIntervalSec、createdAt、updatedAt
玩家狀態players 陣列（nickname、colorIndex、isHost、isOnline）
梯子資料seed（number）、seedSource（UUID hex）、rowCount、colCount、segments 陣列
揭示計數revealedCount 原子計數器
踢除名單kickedPlayerIds Set（同時冗余儲存於 Room JSON）
Session 對應playerId → sessionId Hash

TTL 策略

房間狀態TTL觸發條件
waiting / running / revealing24h建立或任何狀態變更時重設
finished1h轉換至 finished 狀態時設定
所有玩家離線5min (300s)最後一個 WS 連線關閉時 EXPIRE
Pod 重啟維持現有 TTL無連線的房間不重設，自然過期

2. Redis Key Schema

完整鍵清單

Key 模式TypeValueTTL用途
room:{code}String (JSON)序列化的 Room 物件（含 players 陣列）24h房間主狀態
room:{code}:ladderString (JSON)LadderData JSON（seed、segments、results）同主鍵梯子結構與結果
room:{code}:revealedCountString (counter)整數字串，如 "0" 到 "N"同主鍵揭示進度原子計數器
room:{code}:kickedSet{ playerId1, playerId2, ... }同主鍵踢除玩家 ID 集合，用於重連檢查
room:{code}:sessionsHash{ playerId: sessionId }同主鍵追蹤活躍 WebSocket Session

範例鍵

room:ALPHA1
room:ALPHA1:ladder
room:ALPHA1:revealedCount
room:ALPHA1:kicked
room:ALPHA1:sessions

3. Room 物件 JSON 結構

Stage 1: waiting（剛建立，等待玩家加入）

{
  "code": "ALPHA1",
  "status": "waiting",
  "hostId": "player-uuid-001",
  "winnerCount": null,
  "revealMode": "manual",
  "autoRevealIntervalSec": null,
  "kickedPlayerIds": [],
  "createdAt": 1745049600000,
  "updatedAt": 1745049600000,
  "players": [
    {
      "id": "player-uuid-001",
      "nickname": "Alice",
      "colorIndex": 0,
      "isHost": true,
      "isOnline": true,
      "joinedAt": 1745049600000
    }
  ]
}
room:ALPHA1:ladder          → ""（空字串或鍵不存在）
room:ALPHA1:revealedCount   → "0"
room:ALPHA1:kicked          → 空 Set
room:ALPHA1:sessions        → { "player-uuid-001": "sess-abc123" }

Stage 2: running（開始遊戲，seedSource 已生成，梯子尚未生成）

重要：START_GAME 只生成 seedSource（UUID v4）並計算 rowCount，不生成梯子結構。梯子延遲至 BEGIN_REVEAL 時原子生成。room:{code}:ladder 鍵在此階段不存在。
{
  "code": "ALPHA1",
  "status": "running",
  "hostId": "player-uuid-001",
  "winnerCount": 1,
  "revealMode": "manual",
  "autoRevealIntervalSec": null,
  "kickedPlayerIds": ["player-uuid-004"],
  "players": [
    { "id": "player-uuid-001", "nickname": "Alice", "colorIndex": 0,
      "isHost": true, "isOnline": true, "joinedAt": 1745049600000 },
    { "id": "player-uuid-002", "nickname": "Bob", "colorIndex": 1,
      "isHost": false, "isOnline": true, "joinedAt": 1745049675000 },
    { "id": "player-uuid-003", "nickname": "Carol", "colorIndex": 2,
      "isHost": false, "isOnline": false, "joinedAt": 1745049720000 }
  ],
  "createdAt": 1745049600000,
  "updatedAt": 1745049930000
}
room:ALPHA1:ladder          → 不存在（BEGIN_REVEAL 時才創建）
room:ALPHA1:revealedCount   → "0"
room:ALPHA1:kicked          → { "player-uuid-004" }

Stage 3: revealing（揭示進行中，BEGIN_REVEAL 已觸發梯子生成）

房間主鍵 JSON 同 Stage 2，但 status 更新為 "revealing"。

room:ALPHA1:revealedCount   → "2"（已揭示 2 位玩家）
room:ALPHA1:ladder          → (BEGIN_REVEAL 時原子生成，此時已存在)
seed 存在於 Redis，但 ROOM_STATE_FULL 以 LadderDataPublic（省略 seed/seedSource）傳送客戶端。seed 至 END_GAME（status=finished）時才首次對客戶端公開（PRD AC-H03-1, NFR-05）。

Stage 4: finished（全部揭示完畢）

      "result": "lose", "startCol": 2, "endCol": 2, ... }

5. TypeScript 介面（packages/shared）

export type RoomStatus = "waiting" | "running" | "revealing" | "finished";

export interface Player {
  readonly id: string;           // UUID v4
  readonly nickname: string;     // 1-20 chars, sanitized
  readonly colorIndex: number;   // 0-49
  readonly isHost: boolean;      // derived: id === room.hostId
  isOnline: boolean;
  readonly joinedAt: number;     // Unix ms
  result: 'win' | 'lose' | null; // null until revealed
}

export interface LadderSegment {
  readonly row: number;          // 0-indexed
  readonly col: number;          // col <-> col+1 has a rung
}

export interface LadderData {
  readonly seed: number;         // Mulberry32 seed = djb2(seedSource) >>> 0
  readonly seedSource: string;   // UUID v4 hex
  readonly rowCount: number;     // clamp(N*3, 20, 60)
  readonly colCount: number;     // = N
  readonly segments: readonly LadderSegment[];
}

// LadderDataPublic — sent during "revealing" (seed omitted)
export interface LadderDataPublic {
  readonly rowCount: number;
  readonly colCount: number;
  readonly segments: readonly LadderSegment[];
}

export interface PathStep {
  readonly row: number;
  readonly col: number;
  readonly direction: "down" | "left" | "right";
}

export interface ResultSlot {
  readonly playerIndex: number;
  readonly playerId: string;
  readonly startCol: number;
  readonly endCol: number;
  readonly isWinner: boolean;    // win/lose, no named prizes (PRD Out-of-Scope #5)
  readonly path: readonly PathStep[];
}

export interface Room {
  readonly code: string;         // 6-char room code
  title: string | null;          // optional (0-50 chars)
  status: RoomStatus;
  hostId: string;                // mutable: host transfer on 60s disconnect
  players: readonly Player[];    // max 50, incl. offline
  winnerCount: number | null;    // W (1 <= W <= N-1); null until set
  ladder: LadderData | null;
  results: readonly ResultSlot[] | null;
  revealedCount: number;
  revealMode: "manual" | "auto";
  autoRevealIntervalSec: number | null;  // 1-30s; null in manual mode
  kickedPlayerIds: readonly string[];
  readonly createdAt: number;
  updatedAt: number;
}

6. 原子操作策略

操作機制說明
建立房間（SETNX）SETNX防重複建立；碰撞時重試最多 10 次，超過 10 次回傳 500
waiting → runningWATCH / MULTI / EXECWATCH room:{code}，MULTI 驗證 N、W 後設定 status=running，EXEC 失敗則 retry
running → revealing（BEGIN_REVEAL）Lua Script原子生成 seed、ladderMap、resultSlots，設定 status=revealing，更新 TTL；不可中斷
REVEAL_NEXT（逐一揭示）INCR原子遞增 room:{code}:revealedCount；INCR 回傳值即新計數，Server 據此廣播 REVEAL_INDEX
REVEAL_ALL_TRIGGER（一鍵全揭）SET原子設定 revealedCount = totalCount（N），廣播 REVEAL_ALL
revealing → finished（END_GAME）MULTI / EXEC原子寫入 status=finished + EXPIRE 1h；seed 首次對客戶端公開
踢除玩家MULTI / EXEC原子更新 Room JSON（移除玩家）+ SADD room:{code}:kicked + EXPIRE 重設
再玩一局Lua Script原子 pruning isOnline=false 玩家、清除 kicked、重置遊戲欄位、設定 status=waiting

8. 查詢模式

操作Redis 命令說明
讀取房間完整狀態GET room:{code}JSON 反序列化為 Room 物件
讀取梯子資料GET room:{code}:ladderJSON 反序列化為 LadderData
讀取揭示計數GET room:{code}:revealedCount字串轉 parseInt
檢查玩家是否被踢SISMEMBER room:{code}:kicked {playerId}O(1) 查找
取得活躍 SessionsHGETALL room:{code}:sessions回傳 playerId → sessionId Map
寫入房間（附 TTL）MULTI / SET / EXPIRE / EXECJSON 序列化後原子寫入

9. 記憶體估算

元件估算大小說明
Room JSON（50 人）~20 KB含 players 陣列（每玩家 ~200B）
LadderData JSON（50人，rowCount=60）~60 KBsegments: N×rowCount/4 ≈ 750 segments × ~80B
ResultSlots（50 人，rowCount=60）~6 KB每 ResultSlot path ~120B × 50
Sessions Hash<1 KB50 entries × ~36B key + ~36B value
Kicked Set<1 KB最多 50 entries × ~36B
每房間合計~88 KB100 房間 ~8.8 MB，遠低於 Redis 記憶體上限
