🗄️ SCHEMA — Redis Schema Design

Version: v1.1 ｜ 2026-04-19 ｜ Based on EDD v1.3 ｜ Redis is the sole persistence layer. There is no SQL database.

1. 資料儲存概覽

持久層選擇

Ladder Room Online 使用 Redis 作為唯一的持久層。選擇 Redis 的原因：

  • 房間生命週期短（數分鐘到數小時），天然適合 TTL 驅動的資料管理
  • WebSocket 廣播需要低延遲讀寫，Redis 的亞毫秒響應完全符合
  • 原子操作（INCR、SETNX、MULTI/EXEC、WATCH）消除競態條件，無需分散式鎖服務
  • 記憶體用量可預測（每房間 ~88 KB，100 房間僅 ~8.8 MB）

TTL 策略

房間狀態TTL觸發條件
waiting / running / revealing24h建立或任何狀態變更時重設
finished1h轉換至 finished 狀態時設定
所有玩家離線5min (300s)最後一個 WS 連線關閉時 EXPIRE

2. Redis Key Schema

完整鍵清單

Key 模式TypeValueTTL用途

Stage 2: running（梯子尚未生成）

重要：START_GAME 只生成 seedSource（UUID v4）並計算 rowCount，不生成梯子結構。梯子在 BEGIN_REVEAL 時原子生成。room:{code}:ladder 鍵在此階段不存在。

{
  "code": "ALPHA1",
  "status": "running",
  "hostId": "player-uuid-001",
  "winnerCount": 1,
  "kickedPlayerIds": ["player-uuid-004"],
  "players": [
    { "id": "player-uuid-001", "nickname": "Alice", "isHost": true, "isOnline": true },
    { "id": "player-uuid-002", "nickname": "Bob", "isHost": false, "isOnline": true },
    { "id": "player-uuid-003", "nickname": "Carol", "isHost": false, "isOnline": false }
  ]
}

Stage 3: revealing（揭示進行中，梯子已生成）

房間主鍵 JSON 同 Stage 2，但 status 更新。room:{code}:revealedCount → "2"（已揭示 2 位玩家）

room:{code}:ladder → BEGIN_REVEAL 時原子生成，此時已存在（含 seed、seedSource、segments、results）。

seed 存在於 Redis，但 ROOM_STATE_FULL 以 LadderDataPublic（省略 seed/seedSource）傳送客戶端。seed 至 END_GAME（status=finished）時才首次對客戶端公開。

Stage 4: finished（全部揭示完畢）

TTL 降為 1h，room:ALPHA1:revealedCount → "3"（等於 players.length）

4. 資料模型（TypeScript → Redis 對應）

interface Room {
  readonly code: string;
  status: 'waiting' | 'running' | 'revealing' | 'finished';
  hostId: string;
  winnerCount: number | null;
  revealMode: 'manual' | 'auto';
  autoRevealIntervalSec: number | null;
  kickedPlayerIds: readonly string[];
  createdAt: number;       // Unix ms
  updatedAt: number;       // Unix ms
  readonly players: readonly Player[];
}

interface Player {
  readonly id: string;
  nickname: string;
  colorIndex: number;
  isHost: boolean;
  isOnline: boolean;
  joinedAt: number;        // Unix ms
}

/**
 * LadderData — 完整梯子資料，含 seed/seedSource
 * 儲存於 Redis room:{code}:ladder
 * 在 BEGIN_REVEAL 時生成（非 START_GAME）
 * 僅在 status=finished 時對客戶端公開（含 seed/seedSource）
 */
interface LadderData {
  readonly seed: number;          // Mulberry32 seed = djb2(seedSource) >>> 0 (uint32)
  readonly seedSource: string;    // UUID v4 hex
  readonly rowCount: number;      // clamp(N*3, 20, 60)
  readonly colCount: number;      // = N
  readonly segments: readonly LadderSegment[];
  readonly results: readonly ResultSlot[];
}

/**
 * LadderDataPublic — 發送給客戶端的公開梯子資料（省略 seed/seedSource）
 * 在 status=revealing 時透過 ROOM_STATE_FULL unicast 給重連玩家
 */
interface LadderDataPublic {
  readonly rowCount: number;
  readonly colCount: number;
  readonly segments: readonly LadderSegment[];
  // seed 與 seedSource 刻意省略
}

interface LadderSegment {
  readonly row: number;
  readonly col: number;             // 橫槓左端的欄位索引（col <-> col+1 has a rung）
}

interface PathStep {
  readonly row: number;
  readonly col: number;
  readonly direction: 'down' | 'left' | 'right';
}

interface ResultSlot {
  readonly playerIndex: number;
  readonly playerId: string;
  readonly startCol: number;
  readonly endCol: number;
  readonly isWinner: boolean;
  readonly path: readonly PathStep[];
}

/** ResultSlotPublic — REVEAL_ALL payload 專用（省略 path 欄位） */
type ResultSlotPublic = Omit<ResultSlot, 'path'>;

TypeScript 欄位 → Redis 鍵對應表

TypeScript 欄位Redis 鍵儲存位置備註
Room.coderoom:{code}JSON 頂層 code 欄位
Room.statusroom:{code}JSON 頂層 status 欄位
Room.hostIdroom:{code}JSON 頂層 hostId 欄位
Room.winnerCountroom:{code}JSON 頂層 winnerCount 欄位number | null
Room.players[]room:{code}JSON 頂層 players 陣列
LadderData.seedroom:{code}:ladderJSON 頂層 seed 欄位revealing 狀態省略；finished 後公開
LadderData.segments[]room:{code}:ladderJSON 頂層 segments 陣列LadderDataPublic 中保留
LadderData.results[]room:{code}:ladderJSON 頂層 results 陣列BEGIN_REVEAL 時原子填入
revealedCountroom:{code}:revealedCountRedis String 計數器原子 INCR；與 Room JSON 同步
kickedPlayerIds（正本）room:{code}:kickedRedis Set 成員SISMEMBER 用於重連檢查
sessionId 對應room:{code}:sessionsRedis Hash field/value

5. 原子操作設計

5.1 建立房間（POST /rooms）

# SET NX EX 原子地設定鍵值與 TTL，確保唯一性
SET room:ALPHA1 "{...roomJson...}" NX EX 86400

# 若成功，原子地初始化 revealedCount
EVAL "
  if redis.call('EXISTS', KEYS[1]) == 0 then return 0 end
  redis.call('SET', KEYS[2], '0', 'EX', ARGV[1])
  return 1
" 2 room:ALPHA1 room:ALPHA1:revealedCount 86400

5.2 開始遊戲（START_GAME）

重要：START_GAME 不生成梯子。梯子在 BEGIN_REVEAL 時才原子生成。

WATCH room:ALPHA1
GET room:ALPHA1  # 讀取並驗證（status=waiting, N>=2, 1<=W<=N-1）

MULTI
  SET room:ALPHA1 "{...status:running, seedSource:uuid, rowCount:N...}"
  DEL room:ALPHA1:revealedCount
  SET room:ALPHA1:revealedCount "0" EX 86400
EXEC  # EXEC null → 重試最多 3 次

5.3 開始揭曉（BEGIN_REVEAL）

應用層計算 LadderData + ResultSlots，Lua Script 原子寫入 ladder/status=revealing，確保「梯子資料」與「狀態=revealing」不可分割。

# Lua Script 確保原子性（偽代碼）
EVAL "
  local room = redis.call('GET', KEYS[1])
  -- 驗證 status=running
  -- 寫入梯子資料（由應用層傳入）
  redis.call('SET', KEYS[2], ARGV[1], 'EX', ARGV[3])  -- ladder
  redis.call('SET', KEYS[1], ARGV[2], 'EX', ARGV[3])  -- room (status=revealing)
  return 1
" 2 room:ALPHA1 room:ALPHA1:ladder {ladderJson} {roomJson} 86400

5.4 逐一揭示（REVEAL_NEXT）

# 1. 原子遞增計數器
INCR room:ALPHA1:revealedCount  → newCount

# 2. 以 WATCH+MULTI/EXEC 同步 Room JSON 快照並 PUBLISH
WATCH room:ALPHA1
MULTI
  SET room:ALPHA1 "{...revealedCount:newCount...}"
  # 備注：PUBLISH 不在 MULTI/EXEC 中；改用 EXEC 後立即 PUBLISH
EXEC

5.5 結束遊戲（END_GAME）

WATCH room:ALPHA1
MULTI
  SET room:ALPHA1 "{...status:finished...}" EX 3600  # TTL 降為 1h
EXEC
# EXEC 成功後廣播含 seed 的 ROOM_STATE
