SCHEMA — Ladder Room Online Redis Schema Design

Version: v1.1   Date: 2026-04-19   Based on: EDD v1.3

Redis 是唯一的持久層，沒有 SQL 資料庫。所有狀態以明確 TTL 儲存於 Redis 鍵值結構中。

1. 資料儲存概覽

持久層選擇

Ladder Room Online 使用 Redis 作為唯一的持久層。選擇 Redis 的原因：

  • 房間生命週期短（數分鐘到數小時），天然適合 TTL 驅動的資料管理
  • WebSocket 廣播需要低延遲讀寫，Redis 的亞毫秒響應完全符合
  • 原子操作（INCR、SETNX、MULTI/EXEC、WATCH）消除競態條件，無需分散式鎖服務
  • 記憶體用量可預測（每房間 ~88 KB，100 房間僅 ~8.8 MB）

儲存的資料類型

類別儲存內容
房間狀態status、hostId、winnerCount (number | null)、revealMode、autoRevealIntervalSec、createdAt、updatedAt
玩家狀態players 陣列（nickname、colorIndex、isHost、isOnline）
梯子資料seed (number)、seedSource (UUID hex)、rowCount、colCount、segments 陣列
揭示計數revealedCount 原子計數器
踢除名單kickedPlayerIds Set（同時冗余儲存於 Room JSON）
Session 對應playerId → sessionId Hash

TTL 策略

房間狀態TTL觸發條件
waiting / running / revealing24h建立或任何狀態變更時重設
finished1h轉換至 finished 狀態時設定
所有玩家離線5min (300s)最後一個 WS 連線關閉時 EXPIRE
Pod 重啟維持現有 TTL無連線的房間不重設，自然過期

所有 room:{code}:* 子鍵與主鍵使用相同的 TTL，並在主鍵更新時一起重設。

序列化策略

  • 複雜物件（Room、LadderData）序列化為 JSON 字串，儲存在 Redis String 鍵
  • revealedCount 使用 Redis 原生計數器（INCR），值為整數字串如 "3"
  • kickedPlayerIds 使用 Redis Set，成員為 playerId 字串
  • sessions 使用 Redis Hash，field 為 playerId，value 為 sessionId

2. Redis Key Schema

Key 模式TypeValueTTL用途
room:{code}String (JSON)序列化的 Room 物件（含 players 陣列）24h（每次狀態變更重設）房間主狀態
room:{code}:ladderString (JSON)LadderData JSON（seed、segments、results）與主鍵同步梯子結構與結果
room:{code}:revealedCountString (counter)整數字串，如 "0" 到 "N"與主鍵同步揭示進度原子計數器
room:{code}:kickedSet{ playerId1, playerId2, ... }與主鍵同步踢除玩家 ID 集合，用於重連檢查
room:{code}:sessionsHash{ playerId: sessionId }與主鍵同步追蹤活躍 WebSocket Session

鍵格式說明

  • {code} 為 6 位大寫英數字房間代碼，例如 ALPHA1
  • 所有子鍵（:ladder、:revealedCount、:kicked、:sessions）的 TTL 與主鍵 room:{code} 一致
  • 主鍵更新時，使用 MULTI/EXEC 批次重設所有子鍵的 EXPIRE
room:ALPHA1
room:ALPHA1:ladder
room:ALPHA1:revealedCount
room:ALPHA1:kicked

// room:ALPHA1:ladder → 不存在（BEGIN_REVEAL 時才創建）
// room:ALPHA1:revealedCount → "0"
// room:ALPHA1:kicked → { "player-uuid-004" }
// room:ALPHA1:sessions → { "player-uuid-001": "sess-abc123", "player-uuid-002": "sess-def456" }

Stage 3: revealing（揭示進行中，BEGIN_REVEAL 已觸發）

房間主鍵 JSON 同 Stage 2，但 status 更新為 "revealing"。room:ALPHA1:revealedCount 實時遞增（例如 "2" 表示已揭示 2 位）。

room:ALPHA1:ladder 在 BEGIN_REVEAL 時原子生成，此時已存在（含 seed、seedSource、segments、results）。seed 存在於 Redis，但以 LadderDataPublic（省略 seed/seedSource）傳送客戶端，直到 END_GAME 後才公開。

Stage 4: finished（全部揭示完畢）

此時 TTL 降為 1h，room:ALPHA1:revealedCount 等於 players.length。room:ALPHA1:ladder 中的 seed 與 seedSource 首次對客戶端公開（ROOM_STATE 廣播含完整 results[]）。

4. TypeScript 資料模型

interface Room {
  readonly code: string;
  status: 'waiting' | 'running' | 'revealing' | 'finished';
  hostId: string;
  winnerCount: number | null;
  revealMode: 'manual' | 'auto';
  autoRevealIntervalSec: number | null;
  kickedPlayerIds: readonly string[];
  createdAt: number;   // Unix ms
  updatedAt: number;   // Unix ms
  readonly players: readonly Player[];
}

interface Player {
  readonly id: string;
  nickname: string;
  colorIndex: number;
  isHost: boolean;
  isOnline: boolean;
  joinedAt: number;    // Unix ms
}

/** 完整梯子資料，含 seed/seedSource；僅在 status=finished 時對客戶端公開 */
interface LadderData {
  readonly seed: number;          // Mulberry32 seed = djb2(seedSource) >>> 0 (uint32)
  readonly seedSource: string;    // UUID v4 hex；START_GAME 時生成，用於審計（PRD FR-03-1）
  readonly rowCount: number;      // clamp(N*3, 20, 60)
  readonly colCount: number;      // = N (所有玩家，含離線)
  readonly segments: readonly LadderSegment[];
  readonly results: readonly ResultSlot[];
}

/** 發送給客戶端的公開梯子資料（省略 seed/seedSource） */
interface LadderDataPublic {
  readonly rowCount: number;
  readonly colCount: number;
  readonly segments: readonly LadderSegment[];
}

interface LadderSegment {
  readonly row: number;
  readonly col: number;   // 橫槓左端欄位索引（col ↔ col+1 有橫槓）
}

interface PathStep {
  readonly row: number;
  readonly col: number;
  readonly direction: 'down' | 'left' | 'right';
}

interface ResultSlot {
  readonly playerIndex: number;
  readonly playerId: string;
  readonly startCol: number;
  readonly endCol: number;
  readonly isWinner: boolean;
  readonly path: readonly PathStep[];  // length = rowCount
}

/** REVEAL_ALL payload 專用（省略 path，避免超過 64KB WS maxPayload） */
type ResultSlotPublic = Omit<ResultSlot, 'path'>;

TypeScript 欄位 → Redis 鍵對應表

5. 原子操作設計

5.1 建立房間（POST /rooms）

並發請求可能生成相同的 6 位房間代碼，必須用 SET NX EX 確保唯一性。

# SET NX EX 原子地設定鍵值與 TTL，確保唯一性
# 若回傳 nil（已存在），重新生成代碼並重試，最多重試 10 次
SET room:ALPHA1 "{...roomJson...}" NX EX 86400

# 成功後原子初始化 revealedCount（使用 Lua 確保原子性）
EVAL "
  if redis.call('EXISTS', KEYS[1]) == 0 then return 0 end
  redis.call('SET', KEYS[2], '0', 'EX', ARGV[1])
  return 1
" 2 room:ALPHA1 room:ALPHA1:revealedCount 86400

# room:ALPHA1:kicked 不預先初始化（SISMEMBER 對不存在的鍵返回 0，行為正確）

5.2 開始遊戲（START_GAME）

重要：START_GAME 不生成梯子（LadderData、ResultSlots）。梯子在 BEGIN_REVEAL 時才原子生成（PRD AC-H03-1, NFR-05）。

WATCH room:ALPHA1
GET room:ALPHA1
# 驗證 status=waiting，請求者是 host，N>=2，1<=W<=N-1
# 應用層生成 seedSource = crypto.randomUUID()；計算 rowCount = clamp(N*3, 20, 60)

MULTI
  SET room:ALPHA1 "{...status:running, seedSource:uuid, rowCount:N, ladder:null...}"
  SET room:ALPHA1:revealedCount "0"
  DEL room:ALPHA1:ladder          # 若有舊梯子資料，確保清除
  EXPIRE room:ALPHA1 86400
  EXPIRE room:ALPHA1:revealedCount 86400
  EXPIRE room:ALPHA1:kicked 86400
  EXPIRE room:ALPHA1:sessions 86400
EXEC
# 廣播 ROOM_STATE（status:running, rowCount）— 不含 seed、不含 ladder

5.3-B 開始揭示（BEGIN_REVEAL）— Lua Script 原子操作

必須同時生成並寫入 LadderData（含 seed、segments）和 ResultSlots，並將 status 更新為 revealing。

# 應用層（GameService）執行純計算（零 I/O）：
#   ladder = GenerateLadder(room.seedSource, N)
#   results = ComputeResults(ladder, winnerCount, rng)

EVAL "
  local room = cjson.decode(ARGV[1])
  room.status = 'revealing'
  redis.call('SET', KEYS[1], cjson.encode(room))
  redis.call('SET', KEYS[2], ARGV[2])   -- 寫入 LadderData JSON（含 seed）
  redis.call('SET', KEYS[3], '0')       -- 重置揭示計數器
  redis.call('EXPIRE', KEYS[1], 86400)
  redis.call('EXPIRE', KEYS[2], 86400)
  redis.call('EXPIRE', KEYS[3], 86400)
  return 1
" 3 room:ALPHA1 room:ALPHA1:ladder room:ALPHA1:revealedCount {roomJson} {ladderDataJson}

# seed 雖已存入 Redis，但不對客戶端公開（直到 END_GAME）

5.4 揭示計數（REVEAL_NEXT）

# 原子遞增，返回新值（本次揭示的 1-based index）
INCR room:ALPHA1:revealedCount
# 返回值如 "3" 表示第 3 位玩家揭示完成

# REVEAL_NEXT 後不自動轉 finished（PRD AC-H04-4）
# 狀態轉換（revealing → finished）需由 Host 另行發送 END_GAME

5.4-B 結束本局（END_GAME）

WATCH room:ALPHA1
GET room:ALPHA1
# 驗證 status=revealing，revealedCount==totalCount，請求者是 host

MULTI
  SET room:ALPHA1 "{...room with status:'finished', updatedAt:now...}"
  EXPIRE room:ALPHA1 3600           # TTL 降至 1h
  EXPIRE room:ALPHA1:ladder 3600
  EXPIRE room:ALPHA1:revealedCount 3600
  EXPIRE room:ALPHA1:kicked 3600
  EXPIRE room:ALPHA1:sessions 3600
EXEC
# 成功後廣播 ROOM_STATE（status:'finished'，含 seed + 完整 results[]）
# seed 至此首次對客戶端公開

5.5 踢除玩家（KICK_PLAYER）

WATCH room:ALPHA1
GET room:ALPHA1
# 驗證請求者是 host 且目標玩家存在

MULTI
  SET room:ALPHA1 "{...roomJson with player removed from players array...}"
  SADD room:ALPHA1:kicked "player-uuid-003"
  EXPIRE room:ALPHA1 86400
  EXPIRE room:ALPHA1:kicked 86400
EXEC

5.6 再玩一局（PLAY_AGAIN）

流程：驗證 status=finished → 計算 onlinePlayers → 若 onlinePlayers.length < 2 → INSUFFICIENT_ONLINE_PLAYERS → 若 winnerCount >= onlinePlayers.length → winnerCount = null → kickedPlayerIds 清空。

WATCH room:ALPHA1
GET room:ALPHA1
# 篩選 onlinePlayers，調整 winnerCount

MULTI
  SET room:ALPHA1 "{...status:'waiting', players:onlinePlayers, winnerCount:adjusted, kickedPlayerIds:[], revealMode:'manual', autoRevealIntervalSec:null, seedSource:null...}"
  DEL room:ALPHA1:ladder            # 清除梯子（DEL 確保鍵完全移除）
  SET room:ALPHA1:revealedCount "0"
  DEL room:ALPHA1:kicked            # 清空踢除 Set（PRD AC-H07-5）
  EXPIRE room:ALPHA1 86400
  EXPIRE room:ALPHA1:revealedCount 86400
  EXPIRE room:ALPHA1:sessions 86400
EXEC
# 成功後廣播 ROOM_STATE（status:'waiting'）

6. 記憶體估算

估算基準：50 名玩家、梯子 10 列 × 50 欄（約 600 個橫槓 segments）、每位玩家 path 長度 10（rowCount）。

8. 查詢模式

8.1 加入房間流程

# Step 1: 讀取房間狀態
GET room:ALPHA1
# 若 nil → 房間不存在，回傳 404

# Step 2: 檢查玩家是否被踢除（O(1)）
SISMEMBER room:ALPHA1:kicked "player-uuid-003"
# 若返回 1 → 玩家已被踢，回傳 403

# Step 3-4: 讀取、修改、原子寫入
WATCH room:ALPHA1
MULTI
  SET room:ALPHA1 "{...roomJson with new player appended...}"
  HSET room:ALPHA1:sessions "player-uuid-005" "sess-new-xyz"
  EXPIRE room:ALPHA1 86400
  EXPIRE room:ALPHA1:sessions 86400
EXEC

8.2 重連查詢

# 一次取得所有需要的狀態
SISMEMBER room:ALPHA1:kicked "player-uuid-002"   # O(1)
GET room:ALPHA1                                   # O(1)
GET room:ALPHA1:ladder                            # O(1)
GET room:ALPHA1:revealedCount                     # O(1)

# 更新 session 與 isOnline 狀態
HSET room:ALPHA1:sessions "player-uuid-002" "sess-new-reconnect"
WATCH room:ALPHA1
MULTI
  SET room:ALPHA1 "{...with player.isOnline=true...}"
  EXPIRE room:ALPHA1 86400
  EXPIRE room:ALPHA1:ladder 86400
  EXPIRE room:ALPHA1:revealedCount 86400
  EXPIRE room:ALPHA1:kicked 86400
  EXPIRE room:ALPHA1:sessions 86400
EXEC

8.3 揭示下一位

# 原子遞增，取得本次揭示的 1-based index
INCR room:ALPHA1:revealedCount
# 返回值如 3 → 廣播 REVEAL_INDEX（results[2]，含完整 path）

# 廣播揭示事件
PUBLISH room:ALPHA1:events "{type:'REVEAL_INDEX', payload:{...}}"

# 注意：不自動轉 finished，等待 Host 的 END_GAME（PRD AC-H04-4）

9. TTL 管理策略

情境TTL設定時機套用對象
房間建立86400s（24h）POST /rooms 成功後所有 room:{code}:* 鍵
狀態變更（任何）86400s（24h）每次 MULTI/EXEC 成功後所有 room:{code}:* 鍵
轉換至 finished3600s（1h）END_GAME 的 EXEC 後所有 room:{code}:* 鍵
所有玩家離線300s（5min）最後一個 WS close 事件後所有 room:{code}:* 鍵
Pod 重啟後重連不重設 → 重連時重設 24h有玩家重連時所有 room:{code}:* 鍵

TTL 管理原則

  1. 子鍵 TTL 與主鍵同步：每次寫入操作在 MULTI/EXEC 中一次重設所有相關鍵的 EXPIRE
  2. finished 狀態降低 TTL：遊戲結束後資料僅供短時間查閱，1h 後自動清理
  3. 空房縮短 TTL：所有玩家離線 5 分鐘後自動清理，防止殭屍房間積累記憶體
  4. Pod 重啟不影響現有 TTL：Redis 鍵的 TTL 在 Pod 重啟後仍然有效，無需掃描重設
  5. TTL 重設在 MULTI/EXEC 內完成：確保原子性，transaction 失敗時 TTL 不會被單獨更新
